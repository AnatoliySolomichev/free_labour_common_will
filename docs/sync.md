# Модуль "Синхронизация"

**Статус:** черновик v0.5
**Назначение документа:** концептуальное описание модуля синхронизации блокчейнов.
Источник истины для проектирования API, реализации и тестов модуля.
Опирается на [`blockchain.md`](blockchain.md) (транспортное ядро) и [`records.md`](records.md) (семантический слой).

---

## 1. Назначение модуля

Модуль обеспечивает **обмен данными между персональными блокчейнами** разных пользователей: доставку блоков, снимков слияния, пломб и (позже) отзывов, а также построение производных данных, которые нельзя вычислить из одной цепи.

Ключевые свойства:

- **Транспортно-независимость.** Логика синхронизации отделена от способа доставки. Агрегатор (HTTP) — одна из реализаций транспорта; в будущем — прямой P2P.
- **Агрегатор — кеш и индекс, не источник истины** (`records.md` §13). Любое его утверждение перепроверяемо обращением к исходной цепи.
- **Агрегатор ничего не подписывает.** Ни одна протокольно значимая операция не требует подписи агрегатора: всё, что он отдаёт, самопроверяемо против со-подписанных корней. Агрегатор — роль, а не сущность; если оператор хочет участвовать (слияние, пломба) — делает это личной цепью, как обычный пользователь. Поэтому вопрос «человек или машина» для агрегатора не стоит: от него нужны ресурсы, а не доверие.
- **Кеш сырого набора участников** — удерживает то, чего нет в зафиксированном снимке (сами листья, пути узлов, блоки-улики), и тем самым разблокирует построение `FraudClaim` (`blockchain.md` §11.9) и окно валидации (§6.5.5).
- **Приватность по умолчанию.** Ветка попадает в синхронизацию только при намеренной публикации/слиянии (`blockchain.md` §8). Передаётся лишь путь к нужной ветке, не всё дерево.

## 2. Границы и связи с модулями

```
records (семантика)          — не зависит от sync
   ▲
   │
blockchain (транспортное ядро) — не зависит от sync
   ▲
   │  зависит от обоих
sync (этот модуль)
   ├── transport (абстракция)
   │     └── aggregator (HTTP) — существующая реализация
   ├── participant cache (§5) — ПЕРВЫЙ СРЕЗ
   ├── merge orchestrator (§6)
   ├── gossip (§7)
   └── discovery (§8)
```

**Направление зависимостей:** `sync` знает про `blockchain` и `records`, но не наоборот. Поэтому оркестрация слияния вызывает `blockchain::MergeSession`, а кеш `sync` наполняется **над** ним — `MergeSession` про кеш ничего не знает.

## 3. Терминология

| Термин | Описание |
|---|---|
| **Транспорт** | Абстракция доставки данных между узлами. Реализации: агрегатор (HTTP), P2P (позже). |
| **Агрегатор** | Узел-кеш/индекс, собирающий цепи и производные данные. Любой пользователь в меру ресурсов. |
| **Кеш участников** | Хранилище сырого набора: листья снимка, пути узлов, блоки-улики. |
| **Композиция** | Запись `parent_root → (left_child_root, right_child_root)` иерархического дерева Меркла снимка. |
| **Свежесть** | Насколько недавно узел получал обновления (снимки/пломбы/отзывы) о данной цепи. |

## 4. Архитектура: транспортно-независимый слой

Синхронизация формулируется поверх абстрактного транспорта `ITransport` — набор операций «опубликовать / получить / получить манифест». Конкретный способ (HTTP-агрегатор, P2P) реализует этот интерфейс.

Концептуально (сигнатуры уточняются при реализации соответствующего среза):

```
ITransport (абстракция):
  publish_block(Block)                 — выложить блок в сеть (gossip)
  publish_snapshot(root, MergeSnapshot, leaves…)  — выложить снимок и его набор
  publish_seal(Seal)
  fetch_block(hash | address) -> Block?
  fetch_snapshot(root) -> …?
  manifest() -> [hash…]                — для pull-диффа
```

Существующий `aggregator` уже реализует pull-обмен блоками (`manifest` + докачка) и submit-эндпоинт — это ядро будущей `AggregatorTransport`. Полная привязка транспорта — отдельный срез после кеша.

Транспорту нужны две категории операций: **публикация/выборка** (gossip, эскиз выше) и **доставка диалога** — упорядоченная передача непрозрачных байтовых сообщений между двумя узлами для bilateral-протокола слияния. `MergeDialogue` (§6.1) потребляет только вторую и ни одного сетевого типа не знает.

### 4.1. Доставка диалога: `IDialogueChannel` и релей (проект)

Минимальный интерфейс второй категории — доставки непрозрачных сообщений `MergeDialogue` (сигнатуры уточняются при реализации):

```
IDialogueChannel:
  send(session_id, bytes)            — отправить конверт партнёру по сессии
  poll() -> [(session_id, bytes)…]   — забрать входящие по порядку seq (+ новые OFFER)
```

Поверх него — релейный конверт `[session_id, seq, blob]`:

- **`session_id`** — 16 случайных байт, генерирует инициатор вместе с OFFER; пара `(партнёр, session_id)` идентифицирует диалог. Мультиплексирование — просто разные сессии; одна ветка ведёт не больше одного активного диалога (черновик у `MergeSession` один), разные ветки — параллельно.
- **`seq`** — счётчик в пределах направления сессии; приём строго по порядку, дубликаты отбрасываются, повторный `send` того же `(session, seq)` идемпотентен. Это закрывает повторную доставку.
- **`blob`** — байты `MergeDialogue` как есть; релей их не интерпретирует.

Реализация релея на агрегаторе (первый под-срез `AggregatorTransport`, §10.6):

```
POST /dialogue/{to_uid}/{session}   — положить конверт в ящик получателя
GET  /dialogue/{my_uid}/inbox       — свои конверты, ?after=seq для докачки
```

Свойства:

- **Тупая труба.** Релей не подписывает, не интерпретирует и не может незаметно подделать конверты — порча валит диалог в FAILED (§6.1); протокольной истины в ящике нет (§1).
- **Тайм-ауты по активности.** Сессия без новых конвертов дольше локального порога GC-ится вместе с ящиком; заглохший диалог просто бросается — DAG это терпит (`blockchain.md` §11.4). Часы релея — его локальная политика, не протокольное время.
- **Приватность (ограничение среза).** Релей видит метаданные (кто с кем сливается) и содержимое конвертов; шифрование конвертов — возможное расширение, вне этого среза.
- Спам/квоты на ящик — открытый вопрос (§10.7).

CLI поверх канала: `bc merge run --peer UID --via URL [--leaf N]` — инициатор, гоняет диалог до DONE/FAILED; `bc merge serve --via URL [--leaf N]` — ответчик: забирает OFFER из ящика и отвечает указанной веткой. Ручные `merge prepare/create/cosign/finalize` остаются как низкоуровневый путь без сети.

**Важно для первого среза:** кеш участников (§5) не требует готового транспорта. Он наполняется из данных, которые уже проходят при слиянии (`BranchTipInfo` несёт путь узлов и tip-блок), и тестируется автономно. Поэтому его можно строить первым.

## 5. Кеш участников (§11.9) — ПЕРВЫЙ СРЕЗ

Зафиксированный `MergeSnapshot` хранит только обязательство (`merkle_root` + HLL), но **не сами листья**. Кеш восстанавливает недостающее, чтобы можно было построить `FraudProofData` (`blockchain.md` §6.5.6).

### 5.1. Что хранится

Две таблицы:

**Листья** — ключ `leaf_hash` (= `MerkleTree::leaf_hash(ExternalRef)`), значение:
```
LeafRecord {
  ExternalRef        ref;         // (chain, node, block, block_hash) участника
  std::vector<Node>  node_path;   // root..node цепи chain → working_pubkey
  Block              evidence;    // блок-0 участника (прообраз block_hash)
}
```

**Композиции** — ключ `parent_root`, значение:
```
Composition { Hash left_child; Hash right_child; }   // канонический порядок (§6.5.1)
```
Для листового снимка (свежий участник) композиции нет: `parent_root == leaf_hash`, лист самодостаточен.

### 5.2. Наполнение

Кеш наполняет **оркестратор слияния** (§6), не `MergeSession`:

- При слиянии из `BranchTipInfo` партнёра берутся `node_path` (`path`), `evidence` (`tip_block`), `ref` (`tip_address` + `tip_hash`) → `LeafRecord` партнёра. Свой `LeafRecord` строится локально.
- Композиция `union_root → (own_root, partner_root)` записывается по факту слияния (порядок канонизируется как в `MergeSnapshot::merge`).
- Через gossip (§7) кеш пополняется чужими листьями/композициями от агрегаторов.

Правило одного слияния реализовано функцией `record_merge` (§5.5); её зовут и `MergeDialogue` (§6.1), и `bc merge finalize` в CLI. Лист стороны извлекается из её tip только пока её снимок — ещё этот лист (первое слияние); композитный снимок своих листьев не раскрывает, они придут через gossip.

### 5.3. Построение доказательства

По `(target_root, leaf_hash)` кеш собирает `FraudProofData`:

1. **Путь Меркла.** DFS по таблице композиций от `target_root` вниз до `leaf_hash`, накапливая соседей: спускаясь в левого ребёнка — сосед справа (`sibling_is_right = true`), в правого — слева. Если `target_root == leaf_hash` — путь пуст. Нет цепочки композиций до листа → лист не входит в снимок (доказательство построить нельзя).
2. **Лист.** `LeafRecord[leaf_hash]` даёт `ref`, `node_path`, `evidence`.
3. Собрать `FraudProofData{ref, merkle_path, node_path, evidence}` → `Serializer::encode` → байты `proof` для `records::FraudClaim`.

Результат проверяется тем же `blockchain::FraudProof::verify` — построение и проверка сходятся по построению (тест: cache → proof → verify == CONFIRMED/REFUTED_*).

### 5.4. Глубина удержания

Локально кеш держит набор на глубину **окна валидации** автора (§6.5.5, `validated_depth`). Глубже — данные у агрегаторов, отдаются перепроверяемо. Точная политика вытеснения — открытый вопрос (§10.1).

### 5.5. Реализация

`modules/sync`, класс `chainsync::ParticipantCache` (namespace `sync` занят
POSIX-функцией `::sync()` из `<unistd.h>`, поэтому `chainsync`; имя модуля и
библиотеки — `sync`). Первый срез — in-memory, вытеснение/персистентность —
открытый вопрос (§10.1).

API:

- `put_leaf(LeafRecord) → leaf_hash` — ключ всегда пересчитывается как
  `MerkleTree::leaf_hash(ref)`, запись не может разойтись со своим ключом.
- `put_composition(a, b) → parent_root` — порядок детей канонизируется и
  родитель пересчитывается через `MerkleTree::combine`, как в
  `MergeSnapshot::merge`; операция коммутативна.
- `merkle_path(target_root, leaf_hash)` — DFS по композициям (§5.3);
  итеративный, с посещённостью — ромбы DAG обходит, на враждебной таблице не
  зацикливается.
- `build_proof` / `build_proof_bytes` — сборка `FraudProofData` и её CBOR для
  поля `proof` в `records::FraudClaim`.
- `record_merge(cache, own_tip, own_snap, partner_tip, partner_snap)` —
  правило наполнения §5.2 одним завершённым слиянием (снимки — до-слиянные).

Кеш намеренно «глупый»: не валидирует пути узлов и подписи улик при приёме —
судья только `FraudProof::verify`.

**Персистентность.** Конструктор с путём даёт LMDB-хранилище (две таблицы:
`leaves`, `compositions`; write-through, полная загрузка при открытии).
`LeafRecord` сериализуется как `FraudProofData` с пустым путём Меркла —
переиспользуется готовый кодек. Вытеснение не реализовано (§10.1) — хранится
всё. CLI держит кеш в `data_dir/sync_cache`: наполняет при `bc merge
finalize`, показывает `bc cache list`, строит клеймы `bc fraud claim`
(пишет `FraudClaim` в свою цепь только при вердикте `CONFIRMED` — отвергнутый
клейм не имеет веса, сфабрикованный штрафует обвинителя).

## 6. Оркестрация слияния

Оборачивает bilateral-протокол `blockchain.md` §6.4 (`prepare_tip → verify_partner_tip → create_pending → co_sign → finalize`) в обмен сообщениями, вместо ручного hex в CLI. Дополнительно наполняет кеш участников (§5.2).

### 6.1. Диалог слияния (реализовано)

`chainsync::MergeDialogue` (`modules/sync`) — машина состояний **одного** слияния. Транспортно-независима в сильном смысле: вход и выход — непрозрачные байтовые сообщения (`start() → [сообщения]`, `on_message(bytes) → [ответы]`); её может гонять любой канал, доставляющий блобы по порядку. Доставка, тайм-ауты по активности и повторы — забота транспорта (§10.2); заглохший диалог просто бросается — DAG это терпит (`blockchain.md` §11.4).

Формат сообщений — deterministic CBOR (RFC 8949 §4.2.1), конверт `[version, type, payload]`:

| Тип | Payload | Кто шлёт |
|---|---|---|
| `OFFER` | `[tip, snapshot]` — свой `BranchTipInfo` + **до-слиянный** снимок | инициатор |
| `ACCEPT` | то же | ответчик |
| `DRAFT` | 32 байта — хеш своего черновика MERGE-блока | оба |
| `COSIG` | 64 байта — co-подпись черновика партнёра | оба |

Поток: `OFFER → (ACCEPT + DRAFT) → DRAFT → COSIG ⇄ COSIG`. Обе стороны обмениваются именно до-слиянными снимками, поэтому коммитят один и тот же объединённый корень (слияние коммутативно). По завершении каждая сторона финализирует свой MERGE-блок, импортирует путь/tip партнёра (`import_partner_data`) и наполняет кеш (§5.2): оба одно-листовых `LeafRecord` (когда сторона сливается впервые и её снимок ещё лист; композитный снимок своих листьев не раскрывает — они придут через gossip §7) плюс композиция `own_root × partner_root`.

Семантика ошибок как у `FraudProof`: `start()`/`on_message` не бросают — испорченное сообщение, нарушение протокола или неуспешная проверка переводят диалог в `FAILED` (`error()` объясняет), и он замолкает. Неуспех после `create_pending` оставляет сохранённый черновик и выросший снимок — поведение `MergeSession`, откат вне рамок среза.

### 6.2. Что осталось оркестратору

Слой над диалогом: выбор партнёров (discovery §8), привязка к реальной доставке (`AggregatorTransport` §10.6), тайм-ауты/повторы (§10.2).

## 7. Gossip (эскиз, §11.3)

Распространение блоков, снимков, пломб (позже — отзывов) между узлами. Существующий pull-обмен блоками агрегатора — отправная точка. Нужно спроектировать: что именно передаётся о снимках и пломбах, как отображается **свежесть** знания, перепроверка своих пломб при новых отзывах. Открытый вопрос (§10.3).

---

## 8. Discovery партнёров (эскиз, §6.5.4)

Порядок предпочтения при выборе, с кем сливаться: граф экономического обмена → «соседи» → высокий порядок/`fill` → ~10% случайных; на бутстрапе — случайно. Требует данных экономического слоя. Открытый вопрос (§10.4).

---

## 9. Вычисление `timestamp_witnessed` (эскиз, §5.2)

Верхняя граница времени существования блока выводится из графа синхронизаций: если чужой блок со временем T ссылается на наш, наш `timestamp_witnessed ≤ T`. Считается по данным кеша/агрегатора при анализе истории. Открытый вопрос (§10.5).

## 10. Открытые вопросы

### 10.1. Политика вытеснения кеша участников
На какую глубину и по какому критерию удерживать листья/композиции локально; когда обращаться к агрегатору. Связано с `validated_depth` и ресурсами узла.

### 10.2. Доставка диалога слияния
Проект зафиксирован в §4.1 (сессии, идемпотентность по `seq`, тайм-ауты по активности, GC ящика). Остаётся открытым: политика повторной инициации заглохших диалогов (когда и с кем пробовать снова — смыкается с discovery §8) и точные пороги GC.

### 10.3. Gossip снимков/пломб и свежесть (§11.3)
Что передаётся о статусе снимков/пломб/отзывов; как UI показывает свежесть; перепроверка своих пломб при получении новых отзывов.

### 10.4. Источник графа обмена для discovery (§6.5.4)
Как sync получает граф экономических связей (модуль экономики ещё не готов); эвристика бутстрапа.

### 10.5. Расчёт `timestamp_witnessed`
Алгоритм обхода графа синхронизаций и где он живёт (узел vs агрегатор).

### 10.6. Роль и границы `AggregatorTransport`
Как существующий `aggregator` (blocks/hashes/ideas + pull-sync) отображается на `ITransport`; какие таблицы добавить для снимков/композиций/пломб. Релей диалога (§4.1) — первый под-срез.

### 10.7. Спам и квоты на ящик релея
Ящик диалога — ресурс агрегатора, нужны квоты. Кандидат — приоритет по накопленной ценности цепи (история обменов, `economy.md` §6): свежая цепь получает малую квоту, цепь с историей — большую; спам дорожает без цензуры. Чёрный список — локальная политика ресурсов оператора, не протокольная истина; глобальных списков не вводим.

## 12. История изменений

- **v0.5** — спроектирована доставка диалога (§4.1): `IDialogueChannel`, релейный конверт `[session, seq, blob]`, ящик-релей на агрегаторе, тайм-ауты по активности, идемпотентность по `seq`; CLI-команды `bc merge run`/`bc merge serve` (проект). Зафиксирован принцип «агрегатор ничего не подписывает» (§1). §10.2 сужен до повторной инициации и порогов GC; добавлен §10.7 (спам/квоты по ценности). Заведён черновик `economy.md` (идентичность через труд, позитивное свидетельствование).
- **v0.4** — кеш участников стал персистентным (LMDB, write-through; `LeafRecord` хранится как `FraudProofData` с пустым путём); правило наполнения §5.2 вынесено в `record_merge` (общее для `MergeDialogue` и CLI). CLI: `bc merge finalize` наполняет кеш (`data_dir/sync_cache`), новые команды `bc cache list` и `bc fraud claim` (пишет `FraudClaim` только при `CONFIRMED`). Закрыт §11.9 `blockchain.md` в части хранилища/API.
- **v0.3** — реализован диалог слияния: `chainsync::MergeDialogue` (§6.1) — bilateral-протокол §6.4 как транспортно-независимая байтовая машина состояний (deterministic-CBOR конверты OFFER/ACCEPT/DRAFT/COSIG), наполнение кеша участников по §5.2. §10.2 сужен до вопросов доставки. Тесты: полное слияние двух узлов с проверкой кешей обеих сторон, рост DAG при втором слиянии, отказы (пустые ветки, порча сообщений/co-подписи, нарушение порядка).
- **v0.2** — реализован первый срез: `chainsync::ParticipantCache` в `modules/sync` (§5.5) — таблицы листьев/композиций, DFS-сборка пути Меркла, `build_proof`/`build_proof_bytes`; сквозные тесты cache → proof → `FraudProof::verify` (одиночный лист, парное слияние, иерархический DAG, ромб).
- **v0.1** — первая версия. Зафиксированы: транспортно-независимая архитектура (агрегатор — одна из реализаций), кеш участников (§5) как первый срез, эскизы оркестрации/gossip/discovery/`timestamp_witnessed`, открытые вопросы §10.
