# Реестр идей разработчика (ИР)

**Статус документа:** реестр идей-направлений с собственной сквозной нумерацией `ИР-NNN`, отдельной от открытых вопросов модулей (`economy.md` §8, `blockchain.md` §11, `records.md` §14). Идея попадает сюда **до** того, как станет решением: у каждой — статус (**идея** / **рекомендация** / **дискуссионная** / **принята** / **отклонена**) и ссылки на затронутые разделы. Принятая идея переносится решением в профильный документ; здесь остаётся строка со ссылкой. Номера не переиспользуются.

---

## ИР-001. «Лицензия эмиссии» — эмиссия только с высокоприоритетных узлов

**Статус:** идея (2026-07-11).

**Проблема:** украденный ключ глубокой (тёплой) ветки сегодня может самоэмитировать часы — вгонять владельца в долг. Нить эмиссии (`economy.md` §4.3) ловит эквивокацию, но не запрещает саму эмиссию вором, продолжившим нить честно.

**Суть:** право эмиссии — делегируемое сверху вниз, как ключи: корень (или высокий узел) записью в своей ветке разрешает эмиссию узлу/поддереву с **лимитом**; с какой-то глубины делегирование дальше запрещено. Листовые (карманные) ключи тратят имеющееся, но не рождают новый долг: кража кармана ≠ кража печатного станка.

**Механика (набросок):** запись `EmissionGrant` в ветке предка (паттерн делегирования/отзыва): (поддерево, лимит, условия продления). Валидация порции самоэмиссии: цепочка грантов вдоль пути + остаток лимита по нити эмиссии.

**Связано:** `economy.md` §4.3, §5а (градиент защиты ценности), `blockchain.md` §4.4 (градиент приоритета), ИР-002, ИР-003.

## ИР-002. «Градиент необратимости» — необратимые действия только приоритетным ключам

**Статус:** идея (2026-07-11).

**Суть:** обобщение ИР-001 и отложенного «крупная трата — с подписью предка» (`economy.md` §8.8): классифицировать типы записей по отменяемости при краже ключа (правило ретроактивности, `blockchain.md` §11.1) и требовать для необратимых (эмиссия, крупные двусторонние сделки, передача собственности) минимальную высоту ключа; полностью отменяемые (реакции, заявления) доступны любым ключам.

**Оговорка:** слияния и пломбы — повседневная работа листовых веток (весь §6 `blockchain.md`); тотальный запрет «необратимого» листьям убил бы граф синхронизации. Порог взвешивать по ценности, а не по формальной необратимости.

**Связано:** `blockchain.md` §11.10 (окно оспаривания), §11.1 (ретроактивность), ИР-001.

## ИР-003. «Казначейские ветки» — выделенные ветки под эмиссию и погашение

**Статус:** рекомендация (2026-07-11).

**Суть:** держать нить эмиссии (`economy.md` §4.3) в одной-двух выделенных ветках средней высоты («казначейство»): эмиссии и квитанции погашения пишутся только там; повседневные ветки пополняются **внутренними переводами** из казначейской. Нить локальна — не прыгает по веткам; проверка контрагентом дешевле (одна ветка вместо обхода дерева); чекпойнт вершины строится тривиально.

**Заметка:** при принятии ИР-001 казначейские ветки возникают сами — это узлы, которым делегирована эмиссия.

**Связано:** `economy.md` §4.3, §5а; ИР-001.

## ИР-004. «Сквозная нить последовательности» — общий порядковый номер блоков цепи

**Статус:** отклонена (2026-07-11) — конфликт с параллельной офлайн-работой веток (базовое свойство дерева); вместо тотальной нити — специализированные нити по классам записей: эмиссия (принята, `economy.md` §4.3), собственность (кандидат при проектировании §8.12). Разбор сохранён ниже.

**Суть:** прошивать сквозной нитью (номер + ссылка на предыдущее звено, как в нити эмиссии) все блоки цепи — либо только «необратимые» (пломбы, соглашения, сделки). Каждый важный блок получает общецепной порядковый номер; вор не может незаметно вести украденную ветку: его звено либо вне нити (веса не имеет), либо форкает нить (эквивокация, ловится агрегаторами при первом же расхождении). Проверка выборочная: кому не надо — не проверяет.

**За:** глобальный порядок действий цепи без часов; мгновенная объективная детекция «параллельной жизни» украденного ключа; обобщение уже принятой нити эмиссии; дёшево на блок (номер + хеш).

**Против:** сериализует цепь — ломает параллельную офлайн-работу веток с разных устройств (телефон и компьютер не могут одновременно писать важные блоки, не согласовав номер — «двойная трата самому себе» на каждом шаге); частичная нить («только важные») всё равно бьёт по самым частым блокам активных веток (слияния, пломбы); пропуски номеров раскрывают объём скрытой активности (цена приватности веток, `blockchain.md` §8).

**Оценка (Claude, 2026-07-11):** тотальная нить — нет: конфликт с параллельностью веток, базовым свойством дерева. Вместо неё — **специализированные нити по классам записей**, где эквивокация опаснее всего: эмиссия (принята, §4.3), собственность (нить `OwnershipTransfer` — кандидат при проектировании §8.12 `economy.md`), возможно крупные переводы. Несколько независимых нитей не сериализуют повседневную работу и дают ту же детекцию там, где это важно.

## ИР-005. «Живой пилот» — приглашение знакомых, самоописание, стыковка потребностей и навыков

**Статус:** реализована (2026-07-13) — конвенция в `records.md` §8.6/§8.7/§8.8, каталоги в `docs/catalogs.md`, пилот работает. Продолжение вынесено в ИР-006 (приоритет), ИР-007, ИР-008.

**Решения (2026-07-13).** (1) Самоописание — **на уровне идей, а не протокола**: теговые `Concept`-ы, без нового дискриминатора (§8.6, решает §14.6). (2) Каталоги профессий/потребностей — **гибрид**: JSON в репозитории + раздача агрегатором, позже опц. референс-цепь (§8.7). (3) Модель профиля — **атомы + опциональная карточка** `Composite`. (4) LLM в проект не встраивается — вместо этого **JSON-экспорт** профилей для внешнего ИИ/ручной стыковки (§13). (5) Писарь/ИИ готовит **черновик** записей (без ключей), хозяин применяет и **подписывает своим ключом** (`bc apply`).

**Суть:** пригласить несколько знакомых завести свои блокчейны и начать взаимодействовать средствами проекта. Каждый вносит в свою цепь **самоописание**: (а) навыки/профессии/хобби; (б) насущные потребности — сейчас и на горизонте нескольких лет; (в) что мешает их закрыть; (г) какие профессии хотел бы освоить; (д) какие отрасли хотел бы развивать; (е) готовность к переобучению. Из собранных цепей — вручную или с подсказками ИИ (локального/коммерческого) — конструируем механизмы взаимодействия: **потребности одних закрываются навыками других**. Опора: базовые потребности у людей во многом одинаковы (еда, одежда, жильё, техника, дети, отдых), а дефицит специальностей закрывается переобучением, к которому люди декларируют готовность.

**Что уже готово (можно пробовать сегодня).** Полный клиентский цикл на одного человека: `identity create`, ветки, запись знаний (`Concept`/`Entity`/`Property`/`Relation`/`Role`) и труда (`Specialty`/`Grade`/`Worker`/`WorkRecord`/`Acceptance`). Экономика: `Transfer`, `Pledge`, `Redemption`, `wallet`. Обмен цепями: `merge run`/`serve` через агрегатор-релей, ручной двусторонний merge, `discover`, `chain info`, `ideas top`, `rates`. Значит: человек **уже** может описать себя записями, а знакомые — обменяться цепями и увидеть друг друга через агрегатор.

**Чего не хватает (с оценкой сложности).**
1. **Конвенция профиля** — ✅ зафиксирована: теговые `Concept`-ы (`kind:*`/`cat:*`/`horizon:*`/`retrain:*`), модель «атомы + опц. карточка» (§8.6). Только документ — **XS**; опциональный хелпер `bc profile skill/need <slug>` поверх `concept add` с валидацией слага — **S**.
2. **Универсальные каталоги** — ✅ реализовано: формат (§8.7, `docs/catalogs.md`), семя `professions.json`/`needs.json`, раздача агрегатором (`--catalog PATH` → `GET /catalog`), `bc catalog --search`, **отказ от неизвестного слага** при записи (опечатка не падает громко — она делает человека ненаходимым), поле `closed_by` («потребность → закрывающая профессия»). (Замена прежней идеи отдельного протокольного типа «Потребность»: спрос выражается `Concept{kind:need}`, §14.7 остаётся про экономическую «заявку исполнителя» — это разные вещи.)
3. **Справочник в агрегаторе** — ✅ реализовано: `ProfileView` + `GET /directory/<facet>/<slug>`, `bc directory --skill <slug>`, `bc needs list [--need <slug>]`. Закрытые факты выпадают. Внешние инструменты участнику больше не нужны.
4. **JSON-экспорт профилей и потребностей** — ✅ реализовано: `GET /profiles`, `bc export profiles` (§13). LLM в проект не встроен: экспорт кормит внешний ИИ либо читается человеком.

4а. **Стыковка `bc match`** — ✅ реализовано (`MatchView`, `GET /match`): потребность → кто закроет (с проверкой досягаемости по `geo`/`r`/`remote`), **кольца обмена** (замкнутая помощь без долга наружу), дефицит и кто готов в него переучиться. Соединяет `ProfileView` + `closed_by` из каталога. Правило: недостающие данные никого не прячут — не указан радиус или координата означает «без ограничений», а не «не подходит». `--json` отдаёт отчёт внешнему ИИ.
5. **Писарь/ИИ готовит — хозяин подписывает** — ✅ реализовано (`records.md` §8.8): `bc apply --draft FILE [--dry-run] [--yes]`, белый список обратимых типов (перевод не спрячется среди фактов профиля), потолок 20 записей, полный рендер, одно осознанное подтверждение.
6. **Дружелюбный онбординг** (веб/моб/форма) для нетехнических участников (электрик, швея, повар). Сырой C++-CLI в Docker им недоступен. Для пилота — обходится: писарь/ИИ готовит черновики (п. 5), хозяин подписывает. Полноценный фронт — **L**, вне текущего модуля блокчейна.

**Пилот запущен вживую (2026-07-13).** Прогон впятером: каждый подписал профиль из черновика, агрегатор собрал выгрузку, стыковка дала **5 закрытых потребностей из 6**, кольцо обмена из трёх человек (швея → электрик → повар → швея) и один дефицит (ремонт техники), совпавший с человеком, заявившим готовность переучиться. Материалы для знакомых: `docs/pilot.md` (шпаргалка), `docs/pilot/scribe-prompt.md` (промпт ИИ-писарю), `docs/pilot/draft-template.json`. Прогон вскрыл два пробела, оба закрыты: в каталоге не было связи «потребность → закрывающая профессия» (поле `closed_by`), и отсутствовала `prof.appliance-repair`.

**Итог (2026-07-13).** Пилот реализован и работает: пункты 1–5 и 4а закрыты, всё проверено на живых прогонах. Осталось **три пробела**, каждый вынесен в отдельную идею, чтобы не потеряться:

- **ИР-006** — сделка идёт мимо проекта (стыковка нашла людей, а расчёт трудочасами с ней не связан). **Приоритет.**
- **ИР-007** — каталог живёт файлом, а не в блоках цепей.
- **ИР-008** — нет интерфейса для нетехнических участников (бывший п. 6).

**Связано:** `records.md` §8.6 (теговая конвенция профиля), §8.7 + `docs/catalogs.md` (каталоги), §8.8 (черновик и подпись), §13 (экспорт профилей), §14.6 (РЕШЕНО), §14.4 (лестница разрядов — доверие к навыку); `sync.md` §8 (discover); ИР-006/007/008 (продолжение); ИР-002 (градиент необратимости — применён в белом списке черновика).

## ИР-006. «Замкнуть круг» — от стыковки к сделке и расчёту трудочасами

**Статус:** дискуссионная (2026-07-13). **Приоритет — первый.**

**Проблема:** `bc match` находит, что проводку Вере закроет Анна, — и на этом проект умолкает. Дальше люди договариваются в мессенджере, а трудовой и экономический слой (`WorkRecord` → `Acceptance` → `Transfer`, уже полностью рабочий) со стыковкой **никак не связан**. Круг «нашли друг друга → сделали → расплатились трудом» не замыкается.

**Три конкретных разрыва (по коду, 2026-07-13):**

1. **Профиль и экономика — разные вселенные.** Профиль: `Concept{cat:prof.electrician, grade:4}`. Экономика: `Specialty{name}` → `Grade{specialty, level}` → `WorkRecord{agent: Ref→Grade}`. У участника пилота есть только концепты и **ни одной** экономической записи: чтобы залогировать труд, Анне надо сперва вручную завести `Specialty` + `Grade`, ничем не связанные с её профилем.

2. **Ставки ключуются свободным текстом.** `RateEntry.specialty` = `Specialty.name` — **глобальный строковый ключ по всей сети** (§11.2). «Электрик», «электрик», «Электромонтёр» дают три разные ставки и рвут статистику. **Слаг каталога — готовый глобальный ключ** (§8.7): каталог, построенный для стыковки, решает заодно и эту проблему. Сильный довод сделать специальность слагом.

3. **Сделка не привязана к потребности.** `WorkRecord` ничем не ссылается на закрываемую потребность, а после оплаты потребность не закрывается сама — связь «закрыто» (§8.6) надо собирать руками. Это же упирается в открытый вопрос `records.md` §14.7 («заявка на исполнение работы»): под найденную потребность может быть несколько желающих, и кто-то должен зафиксировать «беру».

**Направление (набросок):** специальность = слаг каталога (глобальный ключ ставок); запись о принятии заказа (закрывает §14.7); `WorkRecord`/приёмка ссылаются на потребность; после расчёта потребность закрывается автоматически. Возможна сквозная команда `bc deal`, ведущая от найденного матча до закрытой потребности.

**Связано:** `records.md` §9 (труд), §11.2 (ставки), §12 (принципы), §14.7 (ОТКРЫТ — заявка), §8.6/§8.7; `economy.md` §4; ИР-005 (стыковка, откуда всё начинается); ИР-002 (сделка необратима — подпись поштучная, §8.8).

## ИР-007. Каталог в блоках цепей — антиспам выбором источника

**Статус:** идея (2026-07-13). Направление зафиксировано в `docs/catalogs.md`.

**Суть:** сейчас каталог — JSON-файлы в репозитории, которые раздаёт агрегатор. Это стартовый костыль, а не конечная форма: каталог — та же «любая информация», что и профиль, и должен жить в цепях. Запись каталога — обычный `Concept` с тегом `kind:catalog-entry` и слагом; вести его можно в отдельной «идеологической» цепи, без трудочасов. У каждого свой каталог; **защита от спама — выбором источника, а не модерацией**: клиент настраивает, какие цепи признаёт источниками (на старте — «только моя цепь»). Кто засорил свой каталог, засорил только себя — его не выбирают.

**Почему не срочно:** слаги при переезде не меняются, поэтому уже подписанные профили не сломаются. Переход безопасен в любой момент.

**Связано:** `records.md` §8.7, `docs/catalogs.md`; ИР-005; ИР-006 (если специальность станет слагом — каталог становится ещё важнее).

## ИР-008. Дружелюбный интерфейс для нетехнических участников

**Статус:** идея (2026-07-13). Вынесено из ИР-005 п. 6.

**Суть:** электрик, швея, повар не осилят сырой C++-CLI в Docker. Сейчас это обходится черновиком (§8.8): писарь или ИИ готовит записи, человек читает и подписывает. Но для роста за круг знакомых нужен нормальный вход — веб-форма или мобильный клиент, где человек выбирает слаг из каталога, пишет о себе живым текстом и подписывает своим ключом.

**Ограничение, которое нельзя потерять:** ключ остаётся у человека, подпись — его акт. Любой интерфейс, который «подписывает за пользователя» на сервере, уничтожает главное свойство проекта.

**Оценка:** **L**, вне модуля блокчейна; требует отдельного решения по архитектуре (где живёт ключ в браузере/телефоне).

**Связано:** ИР-005 (п. 5, черновик как временный обходной путь); `docs/pilot.md`.
