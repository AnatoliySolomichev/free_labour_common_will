# Модуль "Блокчейн"

**Статус:** черновик v0.1
**Назначение документа:** концептуальное описание модуля блокчейна.
Источник истины для проектирования API, реализации и тестов.
Документ редактируется по мере уточнения требований.

---

## 1. Назначение модуля

Модуль реализует **персональный разветвляющийся блокчейн** — структуру данных, в которой каждый пользователь ведёт собственную историю записей в виде дерева веток, с криптографической защитой от модификации задним числом и механизмом взаимного заверения с другими пользователями.

Ключевые свойства:

- У каждого пользователя — своя независимая структура (нет общей цепи).
- Структура — двоичное дерево из служебных **узлов**; из **любого** узла может расти линейная **ветка** пользовательских **блоков** (v0.7).
- Иерархия ключей с разной степенью защищённости — ближе к корню сильнее.
- Защита от изменения истории через **взаимные подписи** с другими пользователями ("пломбы").
- Возможность массового группового заверения через попарные слияния, образующие растущий граф синхронизации (DAG).

## 2. Терминология

| Термин | English | Описание |
|---|---|---|
| **Узел** | node, fork-node | Служебная сущность — элемент двоичного дерева ветвления. Содержит публичные ключи и подписи, не содержит пользовательских данных. |
| **Корневой узел** | root node | Узел с индексом 0, единственный для пользователя. Его публичный ключ — глобальный идентификатор человека. |
| **Ветка** | branch | Линейная последовательность пользовательских блоков, растущая из любого узла дерева. |
| **Блок** | block | Пользовательский блок в ветке. Содержит произвольные данные. |
| **Блок слияния** | merge block | Специальный тип блока, фиксирующий синхронизацию с веткой другого пользователя (или своей другой веткой). |
| **Путь** | path | Последовательность узлов от корня до интересующей ветки. |
| **Адрес** | address | Уникальный идентификатор блока: `(идентификатор_пользователя, индекс_узла, индекс_блока_в_ветке)`. |
| **Структурный ключ** | structural key | Ключ, определяющий позицию узла в дереве. В MVP совпадает с рабочим. |
| **Рабочий ключ** | working key | Ключ, которым подписываются блоки ветки. В MVP совпадает со структурным. |
| **Пломба** | seal | Подпись чужим пользователем блока (или его хеша), делающая блок неизменяемым. |

## 3. Структура дерева

### 3.1. Параметры

- **Глубина дерева:** 31 уровень (от корня).
- **Максимум узлов:** 2³² − 1 ≈ 4.29 × 10⁹.
- **Узлы создаются лениво** — только те, которые реально нужны для активных веток.

### 3.2. Нумерация узлов

Используется heap-нумерация двоичного дерева:

```
                    [0]
                   /   \
                 [1]   [2]
                /  \   /  \
              [3] [4][5] [6]
              ...
```

Для узла с индексом `N`:
- Левый потомок: `2N + 1`
- Правый потомок: `2N + 2`
- Родитель: `(N - 1) / 2` (целочисленное деление)

Глубина узла N равна `⌊log₂(N + 1)⌋`. Узлы глубины 31 — листья: у них нет потомков-узлов, только ветка.

**Ветка может расти из любого узла** (v0.7, обсуждение 2026-07-09): узел середины дерева имеет и потомков-узлов, и собственную линейную ветку. Семантика вложенности — информация уровня «фирма» в ветке высокого узла, «отдел» — ниже, «человек/канал» — в глубоких ветках; ценность и данные распределяются по дереву соразмерно защите ключей (§4.1).

### 3.3. Свойства узла

Каждый узел содержит:
- Публичный ключ (структурный + рабочий; в MVP они совпадают).
- Хеш родительского узла (для корневого узла — нулевой хеш).
- Индекс узла в дереве (heap-индекс).
- Признак: левый или правый потомок родителя.
- Подпись родительским ключом (для корневого — самоподпись).
- Метку времени создания (UTC, точность до секунды).

Узел сам по себе **не содержит пользовательских данных**.

### 3.4. Адресация

Адрес блока:
```
(root_pubkey, node_index, block_index_in_branch)
```

- `root_pubkey` — публичный ключ корневого узла, идентификатор человека.
- `node_index` — heap-индекс листового узла, к которому относится ветка.
- `block_index_in_branch` — порядковый индекс блока в линейной ветке (0, 1, 2, ...).

Из `node_index` детерминированно восстанавливается путь от корня (последовательность ключей и узлов).

### 3.5. Длина пути

Чтобы добраться до узла на глубине D, нужно пройти D + 1 узлов (включая корень).

Например, для узла с индексом 2¹⁷ путь содержит 18 узлов, проверяется 18 пар ключей.

## 4. Иерархия ключей

### 4.1. Принцип градиента защищённости

- Ключи **корневого узла и ближних к нему узлов** (глубина 0–5) — высшая защита, хранение на изолированных носителях (бумага, аппаратные кошельки).
- Ключи **средних уровней** (глубина 6–20) — стандартная защита, защищённое хранилище ОС.
- Ключи **глубоких узлов** (глубина 21–31) — рабочие, могут храниться на телефонах и в облаке.

### 4.2. Криптосхема: вариант "подпись" (MVP)

Для каждого узла **независимо генерируется** пара Ed25519-ключей. Структурный и рабочий ключ — одна и та же пара. Дочерний узел подписан родительским ключом.

### 4.3. Задел под гибридную схему (будущее)

Структура узла предусматривает **два публичных ключа** в формате — `structural_pubkey` и `working_pubkey`. В MVP они идентичны. В будущем гибрид:
- Структурные ключи — деривированные (BIP32-подобно) из мастер-seed.
- Рабочие ключи — независимые случайные пары, заверенные структурным ключом узла.

Это даст:
- Восстановление структуры дерева из мастер-seed без хранения всех 2³² пар.
- Ротация рабочего ключа без изменения позиции в дереве.
- Раздельное хранение "идентичности" (мастер-seed) и "повседневных подписей" (рабочие ключи).

### 4.4. Градиент приоритета ключей

Приоритет ключей образует **градиент вдоль пути от корня** (§4.1): ключ любого узла полномочен над всем своим поддеревом — правом **отзыва и замены** ключей потомков записью в своей ветке (§5.3, §6.7, §11.1). Чем ближе к корню, тем шире полномочия и холоднее хранение. Повседневная подпись — всегда самым глубоким (наименее приоритетным) ключом ветки; ключи предков достаются только для создания потомков и для отзывов. **Переподписи истории не существует**: изменение задним числом ломает `prev_hash` и чужие пломбы; отзыв и замена действуют вперёд, с указанного блока.

## 5. Блок и ветка

### 5.1. Структура пользовательского блока

| Поле | Описание |
|---|---|
| `address` | Адрес блока (см. §3.4). |
| `prev_hash` | Хеш предыдущего блока в ветке (для блока с индексом 0 — хеш узла своей ветки). |
| `timestamp_claimed` | Время создания, заявленное автором (UTC, секунды). |
| `block_type` | Тип блока: `DATA`, `MERGE`, `KEY_ROTATION`, `REVOCATION`. |
| `payload` | Произвольные данные в формате CBOR. Для блоков типа `DATA` структура payload определяется в [`docs/records.md`](records.md). Может быть открытым или зашифрованным по усмотрению автора. |
| `external_refs` | (Опционально) Список ссылок на блоки других пользователей, упомянутых в этом. |
| `signature` | Подпись блока рабочим ключом узла, к которому относится ветка. |

### 5.2. Подтверждённое время

Помимо `timestamp_claimed`, в системе существует производная характеристика **`timestamp_witnessed`** — самое позднее время, не позднее которого блок точно существовал. Вычисляется по графу синхронизаций: если блок другого пользователя со временем T ссылается на наш блок, наш `timestamp_witnessed ≤ T`.

`timestamp_witnessed` **не хранится в самом блоке**, а вычисляется при анализе истории.

### 5.3. Ветка

Ветка — линейная цепочка блоков от блока с индексом 0 до текущего конца. Каждый блок, кроме первого, ссылается на предыдущий через `prev_hash`. Первый блок (индекс 0) ссылается на хеш узла своей ветки.

**Служебная роль веток предков.** Ветка узла N — естественное место для служебных записей о поддереве N: отзыв ключа потомка пишется блоком в ветке предка (подпись более приоритетным ключом, §4.4) на той высоте, какой аудитории нужно оповещение — запись на уровне 10 видна каждому, кто проверяет любую ветку этого поддерева, эскалация до корня — всей сети. Проверка дешёвая: контрагент уже держит путь узлов (§6.4) и сверяет служебные ветки предков вдоль него. Механизм отзыва — §11.1.

**Внутренние слияния и публичная вершина.** Свои ветки сливаются между собой (§6, слияние «со своей другой веткой») попарно вверх по дереву; наружу выставляется одна **публичная вершина** — ветка высокого узла, чей снимок коммитит состояние всех веток. Внешнее слияние с вершиной фиксирует у партнёров обязательство обо всём дереве — форк любой внутренней ветки конфликтует с ним за один цикл, а внешних слияний нужно меньше.

Ветка существует только если был создан хотя бы один блок. Листовой узел без блоков не считается активной веткой.

### 5.4. Блок-заглушка (bootstrap и якорь времени)

**Блок-заглушка** — обычный `DATA`-блок с пустым payload (CBOR `[]` = `0x80`, «ноль записей»). Пользовательских данных не несёт, существует ради структурной или временно́й роли. Два применения:

1. **Bootstrap для слияния.** Участник слияния обязан иметь хотя бы один блок (§6.4): снимок строится из листьев `H(ExternalRef)`, а `ExternalRef` требует существующего блока-0. Пустая ветка сначала пишет заглушку, затем сливается.
2. **Якорь времени.** После долгого простоя ветки заглушка создаёт свежий кончик; будучи засвидетельствованной через слияние (§5.2, `timestamp_witnessed`), она даёт доказуемую верхнюю границу времени. Вся последующая работа в ветке (через цепочку `prev_hash`) заведомо позже этого момента.

## 6. Операции

### 6.1. Создание идентичности

1. Генерация пары Ed25519 для корневого узла.
2. Создание корневого узла с самоподписью.
3. Сохранение в локальное хранилище.

После этого пользователь существует, но не имеет ни одной ветки.

### 6.2. Создание ветки

Чтобы создать ветку, ведущую к листовому узлу с индексом `N`:

1. Восстановить путь от корня до `N` (последовательность индексов).
2. Для каждого ещё не существующего узла на пути:
   - Сгенерировать новую пару ключей.
   - Создать узел, подписать ключом родителя.
   - Сохранить.
3. После создания всех узлов пути — ветка готова к записи блоков.

### 6.3. Добавление блока в ветку

1. Получить хеш последнего блока в ветке (или хеш листового узла, если блоков ещё нет).
2. Сформировать блок с нужным `payload`, текущим временем, `prev_hash`.
3. Подписать рабочим ключом листового узла ветки.
4. Сохранить.

### 6.4. Слияние двух веток (синхронизация)

**Предусловие:** обе ветки должны иметь хотя бы один блок. Пустая ветка в слиянии не участвует — сначала пишется блок-заглушка (§5.4), затем слияние. Это гарантирует, что у каждого участника есть блок-0 для листа снимка (§6.5.1).

Слияние веток A и B (двух пользователей или двух своих) производится так:

1. Стороны обмениваются минимально необходимой информацией: путь к ветке (узлы), последний блок ветки, его хеш, **и текущий снимок ветки** (§6.5.1).
2. Каждая сторона создаёт у себя `MERGE`-блок:
   - Является следующим блоком в своей ветке.
   - В `payload` содержит: адрес последнего блока партнёра, его хеш, время слияния.
   - Подписан рабочим ключом своей ветки **и** рабочим ключом партнёра.
3. После создания блоков обмен повторяется — каждая сторона передаёт свой подписанный merge-блок другой.

В итоге у каждой стороны в её ветке появляется блок слияния, ссылающийся на состояние партнёра.

### 6.5. Групповое слияние (граф синхронизации, DAG)

Массовое заверение строится **не** как синхронный протокол «все N договорились одновременно», а как **асинхронный рост графа** из двусторонних слияний (§6.4). Предварительная групповая договорённость не требуется.

Базовая единица — bilateral merge (§6.4), он же «слияние порядка 1». Каждое последующее слияние поднимает порядок:

```
Alice ──merge──► Bob                         (порядок 1: Alice+Bob)
                 Bob ──merge──► Carol         (порядок 2: несёт Alice+Bob+Carol)
                                Carol ──► ...  (порядок 3)
```

**Пакет сливается многократно.** Один merge-пакет порядка K может быть взят в слияние любым числом партнёров. Поэтому структура — направленный ацикличный граф (DAG), а не дерево. Ацикличность гарантирована временем: сослаться можно только на уже существующий блок.

#### 6.5.1. Снимок и обязательства

Каждый merge-блок несёт **снимок** охваченного набора участников, неизменный после со-подписи. Следующее слияние строит новый снимок над объединением.

**Хранение и обмен.** Каждая ветка держит свой текущий снимок в локальном хранилище (таблица `snapshots`, ключ `(root_pubkey, node_index)`). При слиянии стороны обмениваются снимками (наравне с путём и tip-блоком, §6.4); каждая объединяет свой снимок с партнёрским (`MergeSnapshot::merge` — иерархический корень + слияние HLL), фиксирует `merkle_root`/`hll_hash` объединения в блоке и удерживает выросший снимок для ветки. Свежий участник (ещё не сливавшийся) выводит снимок из своего кончика: один лист `H(ExternalRef)`.

Merge-блок фиксирует в **подписанной** части три поля-обязательства:

| Поле              | Что фиксирует                                                 | Размер |
|-------------------|--------------------------------------------------------------|--------|
| `merkle_root`     | иерархический корень Меркла над снимком участников            | 32 Б   |
| `hll_hash`        | обязательство о счётчике уникальных участников (HyperLogLog)  | 32 Б   |
| `validated_depth` | само-декларированная глубина полной проверки автором (§6.5.5) | 4 Б    |

**Иерархическая сборка корня.** Корень строится не плоско над всеми листьями, а по DAG: при слиянии пакетов A и B — `merkle_root(parent) = node_hash( merkle_root(A), merkle_root(B) )`. Для свежего участника (ещё не сливавшегося) его «корень» = единственный лист `H(ExternalRef)`. Это даёт:

- проверку композиции «родитель = хеш детей» за **O(1)** на уровень (вместо пересборки над листьями);
- доказательство включения участника = путь вверх по слияниям, O(порядок) хешей.

Лист = `H(ExternalRef)`, где `ExternalRef = (root_pubkey, node_index, block_index, block_hash)` — полный адрес блока-0, а не только идентификатор человека.

- HLL-эскизы **сливаются** поэлементным максимумом: объединение двух пакетов даёт счётчик уникальных участников с автоматической дедупликацией (критично для DAG, где человек может оказаться в нескольких ветвях). Хеш эскиза в блоке не даёт подделать заявленный счётчик.

**Параметры примитивов (протокольные константы):**

- **Дерево Меркла** — доменное разделение в стиле RFC 6962: лист = `BLAKE2b(0x00 ‖ CBOR(ExternalRef))`, внутренний узел = `BLAKE2b(0x01 ‖ left ‖ right)`. Разные префиксы дают защиту от second-preimage (внутренний узел нельзя выдать за лист). Нечётный хвост уровня поднимается без изменений.
- **HLL** — точность `p = 11` (m = 2048 регистров, эскиз 2 КБ, стандартная ошибка ≈ 2.3%). Идентичность участника хешируется по `root_pubkey`; счётчик — стандартная оценка HLL с линейным подсчётом на малых мощностях. `p` **обязан совпадать у всех** — иначе эскизы не сливаются и хеши не сопоставимы. `hll_hash` = `BLAKE2b` массива регистров.

#### 6.5.2. Метаданные пакета (только в кеше)

Сам набор участников и его метаданные **не хранятся в блокчейне** — только в кеше и распространяются gossip. Merge-блок хранит лишь обязательства (§6.5.1). Метаданные:

- `earliest_zero_block`, `latest_zero_block` — временно́е окно набора (по `timestamp_claimed` блоков-0).
- `max_order` — порядок самого крупного вложенного слияния.
- `participant_count` — оценка уникальных участников (из HLL).
- `fill = participant_count / 2^max_order` — коэффициент заполнения: 1.0 = идеально сбалансированное слияние, ниже = «рыхлое» дерево из неравных пар.

`max_order` хранится как `uint32` — жёсткого предела глубины нет (реальную стоимость проверки ограничивает сэмплинг, а не потолок).

В отличие от этих метаданных, `validated_depth` (§6.5.1) — **подписанное** утверждение автора и хранится в блоке, а не в кеше.

#### 6.5.3. Рост графа и выбор ветви

Поскольку пакет сливается многократно, у него несколько «будущих». Участник держит наблюдаемые разветвления в кеше некоторое время, затем выбирает ветвь(и), ведущие к **самому тяжёлому поддереву** — наибольший охват уникальных участников (правило выбора тяжелейшего поддерева, аналог GHOST).

Новые синхроблоки приоритетнее старых: рост идёт вперёд. Старый несведённый пакет не теряется — он связан хешем с новыми (даёт лишь более слабую верхнюю границу `timestamp_witnessed`, §5.2).

#### 6.5.4. Приоритет выбора партнёра

Порядок предпочтения при выборе, с кем сливаться:

1. Партнёры по **графу экономического обмена** (модуль экономики): с кем чаще обмен продукцией и трудочасами — с тем в первую очередь. Естественно устойчиво к ботам.
2. Партнёры «соседей» (с кем сливались те, с кем сливался ты).
3. Более высокий порядок и более высокий `fill`.
4. ~10% — случайные новые участники (защита от замыкания). На этапе бутстрапа, когда истории обмена ещё нет, выбор случайный.

Дополнительный сигнал качества — само-декларированная `validated_depth` пакета (§6.5.5): при прочих равных предпочтительнее сливаться с пакетом, у которого недавние слияния честно заявляют бо́льшую глубину проверки. Вес сигнала масштабируется репутацией декларанта.

Очерёдность инициативы после слияния: приоритет у пакета с бо́льшим хешем; если он не сливается дальше **до появления следующего блока в своей ветке** — приоритет переходит к пакету с меньшим хешем, и так по кругу. Таймаут отсчитывается активностью (появлением блока), а не реальным временем, — общих часов в системе нет.

#### 6.5.5. Верификация: окно само-декларированной глубины

Полная проверка всего снимка вниз до блоков-0 стоит O(2^K) и неограниченна. Вместо жёсткого протокольного минимума применяется **само-декларированная глубина**: автор слияния сам выбирает, на сколько уровней вниз он полностью проверил снимок, и записывает это в `validated_depth` своего merge-блока (подписанное утверждение).

Полная проверка на глубину D означает: для поддерева глубины D под блоком проверены

- (а) подписи блоков-0 — через путь узлов участника до `working_pubkey` (§9.7);
- (б) композиция обязательств на каждом уровне: `merkle_root` родителя = `node_hash` детей (O(1)); слияние HLL-эскизов детей даёт `hll_hash` родителя.

Глубже D — доверие обязательствам, проверенным теми, кто сливал раньше.

**Почему само-декларация, а не фиксированное N.** Жёсткий минимум, который слабый узел не потянул, порождает ложные аттестации. Явно заявленная глубина делает реальное усилие честно видимым: `validated_depth = 0` легально («не проверял, просто со-подписал»), и рынок такой пакет депизиоритизирует (§6.5.4). Это **репутационный сигнал, а не жёсткая гарантия**: завышение ловится, лишь если в окне реально оказался обман и кто-то его проверил, — поэтому вес глубины масштабируется репутацией декларанта плюс случайные глубокие аудиты.

**Перекрывающиеся окна.** Каждый узел слияния попадает в окно всех, кто в следующих `validated_depth` уровней над ним, по всем ветвям DAG, — независимая многократная перепроверка.

**Удержание путей.** Пути узлов участников (`BranchTipInfo.path`) уже передаются при слиянии; они удерживаются в кеше/`external`-хранилище на глубину окна, чтобы `working_pubkey` был доступен для проверки подписей и позднейших обвинений. За пределами окна связку `(chain, node) → working_pubkey` держат агрегаторы и отдают перепроверяемо (модуль синхронизации).

**Каскад ответственности:** со-подписывая слияние, участник ручается за его обязательства в пределах своей `validated_depth`.

#### 6.5.6. Обвинения в обмане

- **Спам** (субъективно): локальный ignore-list в оперативной памяти (фильтр Блума), без следа в цепи, обратимо.
- **Обман в структуре снимка** (объективно проверяемо — `bad_sig` / `hash_mismatch` на конкретном листе): блок-обвинение с приложенным опровержением, включающим **путь узлов участника** (для доступа к `working_pubkey`). Заявленная `validated_depth` — отягчающий фактор: обман в пределах заявленного окна доказывает, что автор либо не проверял, либо смолчал. Реализуется как типизированный `DATA`-блок (тип записи `FraudClaim` в [`records.md`](records.md)), **не** как отдельный core-`BlockType`: репутация относится к семантическому/экономическому слою, а не к криптоинвариантам ядра (§9).

**Асимметрия проверки и построения.** *Проверка* обвинения самодостаточна: `FraudProof::verify(kind, proof, merkle_root)` берёт только обязательство обвиняемого блока и приложенный proof (реализовано; CLI: `bc fraud verify`). А *построение* обвинения требует данных, которых в зафиксированном снимке нет: `MergeSnapshot` хранит лишь обязательство (`merkle_root` + HLL), но **не сами листья**. Чтобы собрать `merkle_path`, нужен полный набор листьев обвиняемого пакета; чтобы собрать `node_path`/`evidence` — дерево узлов и блок-0 участника. Всё это — данные кеш/gossip-слоя (§6.5.2), который удерживается вне ядра. Построение `FraudClaim` откладывается до модуля синхронизации (§11.9).

### 6.6. Ротация рабочего ключа

Производится через специальный блок типа `KEY_ROTATION` в ветке:

1. Сгенерировать новую пару.
2. Создать блок типа `KEY_ROTATION` с новым публичным ключом в `payload`.
3. Подписать **старым рабочим ключом** (чтобы доказать преемственность).
4. Все последующие блоки ветки подписываются новым рабочим ключом.

При валидации цепи валидатор отслеживает блоки `KEY_ROTATION` и переключает ожидаемый ключ.

### 6.7. Отзыв при компрометации

Механизм зафиксирован (обсуждение 2026-07-11; направление — §11.1). Отзыв ключа узла N — блок типа `REVOCATION` в ветке **предка** N, подписанный рабочим ключом ветки предка (по градиенту §4.4 он приоритетнее любого ключа поддерева N).

**Payload (deterministic CBOR):**

| Поле | Описание |
|---|---|
| `revoked_node` | heap-индекс отзываемого узла N |
| `revoked_pubkey` | отзываемый ключ (при ротациях отзывается конкретный ключ) |
| `compromised_since` | заявленный момент компрометации, UTC |
| `replacement` | (опционально) новый рабочий ключ узла N |

**Правила:**

1. **Размещение.** По умолчанию — ветка родителя `(N−1)/2`; выше — по желанию (дубли объединяются) или при компрометации самого родителя (эскалация). Высота записи = радиус попутного оповещения (§5.3); скорость распространения даёт gossip-сертификат (п. 8), полноту — коммит публичной вершины.
2. **Валидность.** Автор блока — предок N (heap-арифметика); подпись верна; `revoked_pubkey` — действующий или исторический ключ N. **Самоотзыв запрещён**: ключ не может отозвать сам себя — вор владеет им наравне с хозяином; добровольная смена без компрометации — `KEY_ROTATION` (§6.6).
3. **Стоп-кран и замена — раздельно.** Первый `REVOCATION` может не содержать `replacement`: экстренная остановка, никаких генераций ключей в панике. Замена назначается позже вторым `REVOCATION` с `replacement` — иначе кошелёк ветки (`economy.md` §5а) был бы заперт навсегда. Действует последнее назначение в порядке ветки предка; при записях с разных высот побеждает более высокая (§4.4).
4. **Граница ретроспективы.** `compromised_since` не может быть раньше последнего собственного коммита публичной вершины, покрывшего ветку N (если коммитов не было — раньше создания узла); заявленный раньше момент обрезается до границы. Коммит — собственная аттестация владения; всё, что до него, и так защищено: блок вора поверх зафиксированного кончика — либо позже коммита, либо форк (самодонос). Граница закрывает «само-стирание» односторонних записей задним числом.
5. **Ретроактивность** («разделение по типу записи», §11.1) для записей, подписанных отозванным ключом:

| Зона | Двусторонние (переводы, слияния, приёмки) | Односторонние (пломбы, реакции, заявления) |
|---|---|---|
| `timestamp_witnessed` ≤ `compromised_since` | стоят | стоят |
| `compromised_since` … публикация отзыва | **стоят** — риск несёт владелец ключа | теряют вес |
| после публикации отзыва | ничтожны | ничтожны |

6. **Поддерево.** Узлы-потомки, созданные ключом N после `compromised_since`, недействительны вместе со своими ветками; созданные ранее — легитимны (их собственные ключи не скомпрометированы). `created_at` узла — заявленное время: анти-backdating создания узлов даёт только свидетельствование финализации (§11.2).
7. **Интервал действия ключа.** У каждого рабочего ключа есть интервал: от назначения (создание узла / `KEY_ROTATION` / `replacement`) до конца (следующая ротация или отзыв). Подпись вне интервала веса не имеет. Сроков жизни (TTL) не вводим — решение 2026-07-11.
8. **Сертификат отзыва** — самопроверяемый пакет: `REVOCATION`-блок + путь узлов от корня до предка-автора; проверяется автономно (подписи пути, предковость, подпись блока). Распространение — склад на агрегаторе по образцу пломб (`sync.md` §7.2, §10.3).
9. **Точка смены ключа и форк** (2026-07-11). Смена отозванной ветки на `replacement` происходит **первым использованием**: первый блок, подписанный заменой, переключает ожидаемый ключ (механика §6.6, только сертификат смены живёт в ветке предка). Блоки старым ключом после точки смены недействительны. При форке отозванной ветки (вор против хозяина) действительно продолжение под ключом замены — она авторизована более высоким ключом (§4.4); реорг локального хранилища не выполняется, конкурирующие версии — улика (`FraudClaim`, форк-самодонос §11.1).
10. **Действительность автора отзыва** (2026-07-11). Запись отзыва учитывается, только если сам автор в момент записи не был под отзывом: записи из подозрительной зоны ветки автора (подписанные скомпрометированным ключом после его `compromised_since` и не подписанные его заменой) игнорируются — отзыв, как одностороннее действие, теряет вес по таблице п. 5. Проверка рекурсивна и конечна (предки строго выше, корень неотзываем). Закрывает «перехват ветки» отзывом от украденного ключа предка; остаточная дыра — backdating `timestamp_claimed` вором — закрывается `timestamp_witnessed` и окном оспаривания (§11.10).
11. **Структура ≠ вес; локальная публикация** (2026-07-11). Структурная валидация ветки не отвергает блоки подозрительных зон (двусторонние записи там юридически стоят) — она лишь следует сменам ключей; классификацию по зонам даёт отдельный запрос статуса (ветка: ACTIVE/FROZEN/REPLACED; блок: CLEAN/SUSPECT/INVALID_AFTER_REPLACEMENT). «Публикация» локально = отзыв присутствует в хранилище проверяющего: правила зон применяются в момент приёма нового (слияние, входящий перевод), а не пересчётом сохранённой истории; полная таблица п. 5 оживает со складом отзывов и `timestamp_witnessed` (модуль синхронизации).

## 7. Подписи (пломбы)

### 7.1. Что подписывается

Подписывающий может:
- **Подписать только хеш блока** ("слепая пломба") — подтверждает существование блока в момент времени, не видит содержимое.
- **Подписать содержимое и хеш** ("открытая пломба") — видит и заверяет содержимое.

Выбор режима — за подписывающим (что прислал автор, то он и видит).

### 7.2. Шифрование payload

Содержимое блока может быть зашифровано автором. В этом случае подписывающий видит зашифрованный payload, может его подписать "вслепую" (фактически подписывая хеш). Позже автор может раскрыть ключ для верификации содержимого третьими сторонами.

### 7.3. Двусторонние и односторонние подписи

- **Двусторонние:** обе стороны соглашаются, обмен подписями (используется при слияниях).
- **Односторонние:** одна сторона копирует публично доступный блок другой и подписывает у себя ("я видел это состояние"). Согласие автора не требуется. Автор может узнать об односторонних пломбах при последующих синхронизациях.

### 7.4. Хранение пломб

Пломбы хранятся **у подписывающего**, и опционально (по обмену) у автора. Глобальный реестр отсутствует. Распространение знания о пломбах — через gossip при синхронизациях.

## 8. Приватность

**Ратифицировано 2026-07-16 (обсуждение ИР-010):** прежняя модель «ветки по умолчанию приватны» отменена в пользу **радикальной публичности**. Мотивы: нить эмиссии и экономический след (кредитная история, держатели бумаги — ИР-010) через приватность не работают; «приватная ветка» ничем не защищена от анонимного слива — в правде сила.

- Записи цепи **по умолчанию публичны**: любой блок может быть опубликован, синхронизирован и прочитан кем угодно; клиенты и агрегаторы вправе строить по ним витрины (справочник, стыковка, экономический след).
- Приватность — **две ручки у человека**, обе применяются до подписи и остаются его личным решением:
  1. **Огрубление** чувствительных полей — точность выбирает автор (координата до района, время до дня).
  2. **Шифрование payload** перед публикацией — блок и его место в цепи публичны, содержимое читают только держатели ключа расшифровки.
- Штатного механизма шифрования в клиенте нет — это **декларированное направление** (решение 2026-07-16): механизм проектируется, когда появится живой запрос участника; до тех пор шифрование — забота автора записи.
- Рекомендации по чувствительным полям (публичность — по умолчанию, но **не принудительна**):

| Чувствительное поле | Рекомендованная ручка |
|---|---|
| координата (`geo`) | огрубить до района (`55.7`, не `55.7512`), компенсировать радиусом `r:` |
| свободный текст потребности (здоровье, семья) | шифровать payload либо не публиковать |
| адресные данные, серийные номера имущества | огрублять или шифровать |

- Цена, принятая осознанно: публичные `geo` + «ему должны много» = карта уязвимости. Ответ — ручки выше, а не возврат к приватным веткам.
- Все ветки по-прежнему структурно одинаковы; при синхронизации передаётся только путь к нужной ветке (не всё дерево узлов) — это вопрос эффективности, не приватности.

## 9. Инварианты

Система должна гарантировать соблюдение следующих инвариантов:

1. **Целостность дерева узлов:** для любого некорневого узла `N` существует узел `(N-1)/2`, и подпись `N` валидна по ключу родителя.
2. **Целостность ветки:** для любого блока с индексом `i > 0` в ветке существует блок с индексом `i-1`, и `prev_hash` соответствует.
3. **Валидность подписей:** все подписи в дереве и ветках проходят криптографическую проверку.
4. **Монотонность времени в ветке:** `timestamp_claimed[i] ≥ timestamp_claimed[i-1]` для всех i. (Соблюдение требует доверия к автору.)
5. **Уникальность позиции:** в ветке не может быть двух блоков с одним индексом.
6. **Уникальность узла:** не может быть двух разных узлов с одним heap-индексом в дереве пользователя.
7. **Связь блока и узла:** блок ветки может быть подписан только рабочим ключом листового узла этой ветки (с учётом возможной ротации).

## 10. Хранилище и сериализация

- **База данных:** LMDB (embedded key-value, транзакционная).
- **Структура таблиц:**
  - `nodes`: ключ = `(root_pubkey, node_index)`, значение = сериализованный узел.
  - `blocks`: ключ = `(root_pubkey, node_index, block_index)`, значение = сериализованный блок.
  - `seals`: ключ = хеш блока, значение = список пломб (подписей других пользователей).
  - `external_blocks`: блоки чужих пользователей, скопированные при синхронизации.
- **Сериализация:** CBOR (RFC 8949) с deterministic encoding (раздел 4.2.1) — гарантирует, что один и тот же логический блок всегда сериализуется в идентичные байты.
- **Хеширование:** BLAKE2b-256.
- **Подписи:** Ed25519.
- **Криптобиблиотека:** libsodium.

## 11. Открытые вопросы

Вопросы, которые отложены до более позднего этапа проектирования:

### 11.1. Размещение блоков отзыва (REVOCATION) — РЕШЕНО (2026-07-11, механизм в §6.7)

Решения 2026-07-11: радиус по умолчанию = родитель (эскалация/дубли выше — по необходимости); стоп-кран и поздняя замена — двумя отдельными `REVOCATION`-блоками; `compromised_since` ограничен последним собственным коммитом публичной вершины; таблица ретроактивности с witnessed-якорем — §6.7. Исходное направление (2026-07-09) ниже.
Отзыв ключа узла N — **блок в ветке предка N** (ветки растут из любых узлов, §3.2/§5.3), подписанный настолько высоким ключом, насколько нужно (§4.4: высший ключ побеждает); высота записи = радиус оповещения. Для скорости распространения — параллельно gossip-сертификат (самопроверяемая копия записи, склад на агрегаторе по образцу пломб). Ретроактивность — по правилу «разделение по типу записи»: двусторонние записи (переводы, слияния, приёмки), совершённые до публикации отзыва, стоят — риск до публикации несёт владелец ключа (как с банковской картой); односторонние (пломбы, реакции, заявления о работе) с `compromised_since` теряют вес. Форк ветки (два подписанных блока с одним адресом) — объективное доказательство компрометации, кандидат в `FraudClaim`; агрегатор обнаруживает коллизию при приёме блока. Детали реализации — при разработке механизма.

### 11.2. Защита от создания фальшивых поддеревьев при компрометации
Если злоумышленник получил рабочий ключ узла N, он может создать новые узлы-потомки и ветки. Варианты защиты:
- Свежесозданные узлы становятся "финальными" только после свидетельствования другими пользователями (до этого — отзываемы родителем без сложностей).
- Двойная подпись при создании узла (родитель + прародитель).

Решение: предположительно — комбинация "свидетельствование как условие финализации" + механизм отзыва родителем. Окончательно решить при проектировании протокола синхронизации.

### 11.3. Распространение информации об отзывах — РЕШЕНО в части распространения (2026-07-11)
Склад сертификатов на агрегаторе + pull-дифф (`sync.md` §7.2), CLI `bc revoke create/publish/fetch --via`. Остаётся (→ `sync.md` §10.3): политика свежести, UI, перепроверка своих пломб при новых отзывах. Исходная постановка ниже.
В модели без глобального реестра распространение происходит через gossip. Нужно спроектировать:
- Какие данные передаются о статусе отзывов при синхронизации.
- Как UI отображает "свежесть" знания об отзывах.
- Перепроверка валидности своих пломб при получении новых отзывов.

### 11.4. Отказоустойчивость группового слияния — РЕШЕНО (v0.3)
В DAG-модели (§6.5) оффлайн-участник перестаёт быть проблемой: заглохшая ветвь не разрушает граф, пакет продолжает расти в других слияниях. Синхронный сбор и предварительная договорённость отменены. Остаётся сетевой вопрос обнаружения партнёров — решается через агрегатор (модуль синхронизации).

### 11.5. Восстановление мастер-идентичности
Направление (2026-07-09): личность **уже отражена** в чужих цепях (двусторонние записи, пломбы, совместный труд) — восстановление = *пере-присоединение отражений*: контрагенты пломбируют «цепь X продолжается как Y» (социальная биометрия — люди помнят человека, без хранения биометрических данных), а поскольку часы именные, каждый эмитент бумаги, которую держала жертва, публично объявляет «мою бумагу, числившуюся за X, гашу для Y» — долги пере-адресуются добровольно, эмитент знает кредитора в лицо. Дополнительно — «завещание»: hash будущего корня в раннем блоке, публикация прообраза при компрометации. Не в MVP; в структуре корневого узла предусмотреть поле «версия мастер-ключа».

### 11.6. Внешнее хранилище больших данных
Для больших payload (видео, файлы) рассмотреть вынос содержимого во внешнее хранилище (по аналогии с IPFS), с хранением в блоке только хеша и метаданных. На текущем этапе — payload хранится непосредственно в блоке.

### 11.7. Ротация ключей структурного уровня
Механизм ротации рабочего ключа описан (§6.6). Ротация структурного ключа (изменение позиции узла в дереве) — отдельная сложная тема, отложена до перехода на гибридную схему ключей.

### 11.8. Каноничность UTF-8 в payload
Если payload содержит текст, нужна нормализация (например, NFC) для каноничности хеширования. Уточнить при первой реализации.

### 11.10. Окно оспаривания — НА БУДУЩЕЕ ОБСУЖДЕНИЕ (2026-07-09)
Единый механизм «запись вступает в силу для третьих лиц через время T, в течение которого более высокий ключ может опротестовать»: для отзывов/реструктуризаций (гонка честного владельца против вора решается приоритетом ключей) и для крупных трат (заявка-намерение, созревающая до покупки; массовые изменения без выдержанных окон — красный флаг для контрагентов). Обсудить: величина T, форма записи-намерения, поведение клиентов.

### 11.9. Кеш набора участников и построение доказательств `FraudClaim`
Зафиксированный снимок (`MergeSnapshot`) хранит только обязательство (`merkle_root` + HLL), но **не сами листья**. Поэтому *проверка* обвинения самодостаточна и уже реализована (`FraudProof::verify`, `bc fraud verify`), а *построение* — нет: чтобы собрать `merkle_path`, нужен полный набор листьев обвиняемого пакета; чтобы собрать `node_path`/`evidence` — дерево узлов и блок-0 участника. Нужно спроектировать (в модуле синхронизации, совместно с §11.3, §11.4):
- **Хранилище сырого набора участников** пакета (листья `ExternalRef`) — на глубину окна валидации (§6.5.5) локально, глубже — у агрегаторов.
- **Удержание путей узлов** участников (`BranchTipInfo.path`) и блоков-0 для доступа к `working_pubkey` и уликам.
- **Gossip-обмен** этими данными и роль агрегатора как перепроверяемого кеша.
- API построения: `bc fraud claim` поверх этого слоя.

**Статус:** закрыто модулем синхронизации (`sync.md` §5) в части хранилища, удержания и API — `chainsync::ParticipantCache` (листья + композиции, персистентно, наполняется при слиянии) и `bc fraud claim` (строит proof из кеша, самопроверяет и пишет `FraudClaim` только при `CONFIRMED`). Остаётся gossip-обмен листьями/композициями и роль агрегатора (`sync.md` §7, §10.3) — без него доказуемы лишь участники, встреченные при собственных слияниях.
