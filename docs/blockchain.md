# Модуль "Блокчейн"

**Статус:** черновик v0.1
**Назначение документа:** концептуальное описание модуля блокчейна.
Источник истины для проектирования API, реализации и тестов.
Документ редактируется по мере уточнения требований.

---

## 1. Назначение модуля

Модуль реализует **персональный разветвляющийся блокчейн** — структуру данных, в которой каждый пользователь ведёт собственную историю записей в виде дерева веток, с криптографической защитой от модификации задним числом и механизмом взаимного заверения с другими пользователями.

Ключевые свойства:

- У каждого пользователя — своя независимая структура (нет общей цепи).
- Структура — двоичное дерево из служебных **узлов**, каждый лист которого продолжается линейной **веткой** пользовательских **блоков**.
- Иерархия ключей с разной степенью защищённости — ближе к корню сильнее.
- Защита от изменения истории через **взаимные подписи** с другими пользователями ("пломбы").
- Возможность массового группового заверения через попарные слияния, образующие растущий граф синхронизации (DAG).

## 2. Терминология

| Термин | English | Описание |
|---|---|---|
| **Узел** | node, fork-node | Служебная сущность — элемент двоичного дерева ветвления. Содержит публичные ключи и подписи, не содержит пользовательских данных. |
| **Корневой узел** | root node | Узел с индексом 0, единственный для пользователя. Его публичный ключ — глобальный идентификатор человека. |
| **Ветка** | branch | Линейная последовательность пользовательских блоков, растущая из листового узла. |
| **Блок** | block | Пользовательский блок в ветке. Содержит произвольные данные. |
| **Блок слияния** | merge block | Специальный тип блока, фиксирующий синхронизацию с веткой другого пользователя (или своей другой веткой). |
| **Путь** | path | Последовательность узлов от корня до интересующей ветки. |
| **Адрес** | address | Уникальный идентификатор блока: `(идентификатор_пользователя, индекс_узла, индекс_блока_в_ветке)`. |
| **Структурный ключ** | structural key | Ключ, определяющий позицию узла в дереве. В MVP совпадает с рабочим. |
| **Рабочий ключ** | working key | Ключ, которым подписываются блоки ветки. В MVP совпадает со структурным. |
| **Пломба** | seal | Подпись чужим пользователем блока (или его хеша), делающая блок неизменяемым. |

## 3. Структура дерева

### 3.1. Параметры

- **Глубина дерева:** 31 уровень (от корня).
- **Максимум узлов:** 2³² − 1 ≈ 4.29 × 10⁹.
- **Узлы создаются лениво** — только те, которые реально нужны для активных веток.

### 3.2. Нумерация узлов

Используется heap-нумерация двоичного дерева:

```
                    [0]
                   /   \
                 [1]   [2]
                /  \   /  \
              [3] [4][5] [6]
              ...
```

Для узла с индексом `N`:
- Левый потомок: `2N + 1`
- Правый потомок: `2N + 2`
- Родитель: `(N - 1) / 2` (целочисленное деление)

Глубина узла N равна `⌊log₂(N + 1)⌋`. Узлы глубины 31 — листья, у них нет потомков-узлов, только линейная ветка.

### 3.3. Свойства узла

Каждый узел содержит:
- Публичный ключ (структурный + рабочий; в MVP они совпадают).
- Хеш родительского узла (для корневого узла — нулевой хеш).
- Индекс узла в дереве (heap-индекс).
- Признак: левый или правый потомок родителя.
- Подпись родительским ключом (для корневого — самоподпись).
- Метку времени создания (UTC, точность до секунды).

Узел сам по себе **не содержит пользовательских данных**.

### 3.4. Адресация

Адрес блока:
```
(root_pubkey, node_index, block_index_in_branch)
```

- `root_pubkey` — публичный ключ корневого узла, идентификатор человека.
- `node_index` — heap-индекс листового узла, к которому относится ветка.
- `block_index_in_branch` — порядковый индекс блока в линейной ветке (0, 1, 2, ...).

Из `node_index` детерминированно восстанавливается путь от корня (последовательность ключей и узлов).

### 3.5. Длина пути

Чтобы добраться до узла на глубине D, нужно пройти D + 1 узлов (включая корень).

Например, для узла с индексом 2¹⁷ путь содержит 18 узлов, проверяется 18 пар ключей.

## 4. Иерархия ключей

### 4.1. Принцип градиента защищённости

- Ключи **корневого узла и ближних к нему узлов** (глубина 0–5) — высшая защита, хранение на изолированных носителях (бумага, аппаратные кошельки).
- Ключи **средних уровней** (глубина 6–20) — стандартная защита, защищённое хранилище ОС.
- Ключи **глубоких узлов** (глубина 21–31) — рабочие, могут храниться на телефонах и в облаке.

### 4.2. Криптосхема: вариант "подпись" (MVP)

Для каждого узла **независимо генерируется** пара Ed25519-ключей. Структурный и рабочий ключ — одна и та же пара. Дочерний узел подписан родительским ключом.

### 4.3. Задел под гибридную схему (будущее)

Структура узла предусматривает **два публичных ключа** в формате — `structural_pubkey` и `working_pubkey`. В MVP они идентичны. В будущем гибрид:
- Структурные ключи — деривированные (BIP32-подобно) из мастер-seed.
- Рабочие ключи — независимые случайные пары, заверенные структурным ключом узла.

Это даст:
- Восстановление структуры дерева из мастер-seed без хранения всех 2³² пар.
- Ротация рабочего ключа без изменения позиции в дереве.
- Раздельное хранение "идентичности" (мастер-seed) и "повседневных подписей" (рабочие ключи).

### 4.4. Приоритет ключей

Ключ родительского узла **может переподписать** действия потомка. Это используется для отзыва скомпрометированной ветки.

## 5. Блок и ветка

### 5.1. Структура пользовательского блока

| Поле | Описание |
|---|---|
| `address` | Адрес блока (см. §3.4). |
| `prev_hash` | Хеш предыдущего блока в ветке (для блока с индексом 0 — хеш листового узла ветки). |
| `timestamp_claimed` | Время создания, заявленное автором (UTC, секунды). |
| `block_type` | Тип блока: `DATA`, `MERGE`, `KEY_ROTATION`, `REVOCATION`. |
| `payload` | Произвольные данные в формате CBOR. Для блоков типа `DATA` структура payload определяется в [`docs/records.md`](records.md). Может быть открытым или зашифрованным по усмотрению автора. |
| `external_refs` | (Опционально) Список ссылок на блоки других пользователей, упомянутых в этом. |
| `signature` | Подпись блока рабочим ключом узла, к которому относится ветка. |

### 5.2. Подтверждённое время

Помимо `timestamp_claimed`, в системе существует производная характеристика **`timestamp_witnessed`** — самое позднее время, не позднее которого блок точно существовал. Вычисляется по графу синхронизаций: если блок другого пользователя со временем T ссылается на наш блок, наш `timestamp_witnessed ≤ T`.

`timestamp_witnessed` **не хранится в самом блоке**, а вычисляется при анализе истории.

### 5.3. Ветка

Ветка — линейная цепочка блоков от блока с индексом 0 до текущего конца. Каждый блок, кроме первого, ссылается на предыдущий через `prev_hash`. Первый блок (индекс 0) ссылается на хеш листового узла дерева.

Ветка существует только если был создан хотя бы один блок. Листовой узел без блоков не считается активной веткой.

## 6. Операции

### 6.1. Создание идентичности

1. Генерация пары Ed25519 для корневого узла.
2. Создание корневого узла с самоподписью.
3. Сохранение в локальное хранилище.

После этого пользователь существует, но не имеет ни одной ветки.

### 6.2. Создание ветки

Чтобы создать ветку, ведущую к листовому узлу с индексом `N`:

1. Восстановить путь от корня до `N` (последовательность индексов).
2. Для каждого ещё не существующего узла на пути:
   - Сгенерировать новую пару ключей.
   - Создать узел, подписать ключом родителя.
   - Сохранить.
3. После создания всех узлов пути — ветка готова к записи блоков.

### 6.3. Добавление блока в ветку

1. Получить хеш последнего блока в ветке (или хеш листового узла, если блоков ещё нет).
2. Сформировать блок с нужным `payload`, текущим временем, `prev_hash`.
3. Подписать рабочим ключом листового узла ветки.
4. Сохранить.

### 6.4. Слияние двух веток (синхронизация)

Слияние веток A и B (двух пользователей или двух своих) производится так:

1. Стороны обмениваются минимально необходимой информацией: путь к ветке (узлы), последний блок ветки, его хеш.
2. Каждая сторона создаёт у себя `MERGE`-блок:
   - Является следующим блоком в своей ветке.
   - В `payload` содержит: адрес последнего блока партнёра, его хеш, время слияния.
   - Подписан рабочим ключом своей ветки **и** рабочим ключом партнёра.
3. После создания блоков обмен повторяется — каждая сторона передаёт свой подписанный merge-блок другой.

В итоге у каждой стороны в её ветке появляется блок слияния, ссылающийся на состояние партнёра.

### 6.5. Групповое слияние (граф синхронизации, DAG)

Массовое заверение строится **не** как синхронный протокол «все N договорились одновременно», а как **асинхронный рост графа** из двусторонних слияний (§6.4). Предварительная групповая договорённость не требуется.

Базовая единица — bilateral merge (§6.4), он же «слияние порядка 1». Каждое последующее слияние поднимает порядок:

```
Alice ──merge──► Bob                         (порядок 1: Alice+Bob)
                 Bob ──merge──► Carol         (порядок 2: несёт Alice+Bob+Carol)
                                Carol ──► ...  (порядок 3)
```

**Пакет сливается многократно.** Один merge-пакет порядка K может быть взят в слияние любым числом партнёров. Поэтому структура — направленный ацикличный граф (DAG), а не дерево. Ацикличность гарантирована временем: сослаться можно только на уже существующий блок.

#### 6.5.1. Снимок и обязательства

Каждый merge-блок несёт **снимок** охваченного набора участников, неизменный после со-подписи. Следующее слияние строит новый снимок над объединением.

Merge-блок фиксирует в цепи два компактных обязательства о наборе:

| Обязательство      | Что фиксирует                                                       | Размер |
|--------------------|---------------------------------------------------------------------|--------|
| **Корень Меркла**  | двоичное дерево хешей над листьями `H(ExternalRef)` каждого блока-0 | 32 Б   |
| **Хеш HLL-эскиза** | обязательство о счётчике уникальных участников (HyperLogLog)        | 32 Б   |

Лист дерева Меркла = `H(ExternalRef)`, где `ExternalRef = (root_pubkey, node_index, block_index, block_hash)` — полный адрес блока-0, а не только идентификатор человека.

- Доказательство «участник X входит в снимок» = путь соседей в дереве Меркла (~log₂N хешей, ≈1 КБ для 4 млрд участников).
- HLL-эскизы **сливаются** поэлементным максимумом: объединение двух пакетов даёт счётчик уникальных участников с автоматической дедупликацией (критично для DAG, где человек может оказаться в нескольких ветвях). Хеш эскиза в блоке не даёт подделать заявленный счётчик.

#### 6.5.2. Метаданные пакета (только в кеше)

Сам набор участников и его метаданные **не хранятся в блокчейне** — только в кеше и распространяются gossip. Merge-блок хранит лишь обязательства (§6.5.1). Метаданные:

- `earliest_zero_block`, `latest_zero_block` — временно́е окно набора (по `timestamp_claimed` блоков-0).
- `max_order` — порядок самого крупного вложенного слияния.
- `participant_count` — оценка уникальных участников (из HLL).
- `fill = participant_count / 2^max_order` — коэффициент заполнения: 1.0 = идеально сбалансированное слияние, ниже = «рыхлое» дерево из неравных пар.

`max_order` хранится как `uint32` — жёсткого предела глубины нет (реальную стоимость проверки ограничивает сэмплинг, а не потолок).

#### 6.5.3. Рост графа и выбор ветви

Поскольку пакет сливается многократно, у него несколько «будущих». Участник держит наблюдаемые разветвления в кеше некоторое время, затем выбирает ветвь(и), ведущие к **самому тяжёлому поддереву** — наибольший охват уникальных участников (правило выбора тяжелейшего поддерева, аналог GHOST).

Новые синхроблоки приоритетнее старых: рост идёт вперёд. Старый несведённый пакет не теряется — он связан хешем с новыми (даёт лишь более слабую верхнюю границу `timestamp_witnessed`, §5.2).

#### 6.5.4. Приоритет выбора партнёра

Порядок предпочтения при выборе, с кем сливаться:

1. Партнёры по **графу экономического обмена** (модуль экономики): с кем чаще обмен продукцией и трудочасами — с тем в первую очередь. Естественно устойчиво к ботам.
2. Партнёры «соседей» (с кем сливались те, с кем сливался ты).
3. Более высокий порядок и более высокий `fill`.
4. ~10% — случайные новые участники (защита от замыкания). На этапе бутстрапа, когда истории обмена ещё нет, выбор случайный.

Очерёдность инициативы после слияния: приоритет у пакета с бо́льшим хешем; если он не сливается дальше **до появления следующего блока в своей ветке** — приоритет переходит к пакету с меньшим хешем, и так по кругу. Таймаут отсчитывается активностью (появлением блока), а не реальным временем, — общих часов в системе нет.

#### 6.5.5. Верификация

Достоверность заявленных `count`/`order`/окна проверяется рекурсивным обходом снимка вниз до блоков-0. Стоимость — O(2^K) на пакет порядка K, поэтому:

- нижние уровни (десятки людей) — проверяются полностью;
- верхние (миллионы) — выборочно: сэмплинг листьев + сверка корня Меркла и HLL.

**Каскад ответственности:** со-подписывая слияние, участник ручается за его обязательства. Уличённый обман бьёт по подписи со-подписанта → есть прямой стимул проверять партнёра до со-подписи.

#### 6.5.6. Обвинения в обмане

- **Спам** (субъективно): локальный ignore-list в оперативной памяти (фильтр Блума), без следа в цепи, обратимо.
- **Обман в математике** (объективно проверяемо — ложный корень / `count` / `order`): блок-обвинение с приложенным опровержением. Реализуется как типизированный `DATA`-блок (тип записи `FraudClaim` в [`records.md`](records.md)), **не** как отдельный core-`BlockType`: репутация относится к семантическому/экономическому слою, а не к криптоинвариантам ядра (§9).

### 6.6. Ротация рабочего ключа

Производится через специальный блок типа `KEY_ROTATION` в ветке:

1. Сгенерировать новую пару.
2. Создать блок типа `KEY_ROTATION` с новым публичным ключом в `payload`.
3. Подписать **старым рабочим ключом** (чтобы доказать преемственность).
4. Все последующие блоки ветки подписываются новым рабочим ключом.

При валидации цепи валидатор отслеживает блоки `KEY_ROTATION` и переключает ожидаемый ключ.

### 6.7. Отзыв при компрометации

Если ключ узла N скомпрометирован, владелец использует ключ узла-родителя `(N-1)/2`:

1. Создать блок типа `REVOCATION` (механизм размещения — открытый вопрос, см. §11).
2. Указать в нём: индекс отзываемого узла, время компрометации, новый публичный ключ для замены.
3. Подписать ключом родительского узла.

После этого все блоки ветки и потомков узла N, созданные после времени компрометации, считаются недействительными.

## 7. Подписи (пломбы)

### 7.1. Что подписывается

Подписывающий может:
- **Подписать только хеш блока** ("слепая пломба") — подтверждает существование блока в момент времени, не видит содержимое.
- **Подписать содержимое и хеш** ("открытая пломба") — видит и заверяет содержимое.

Выбор режима — за подписывающим (что прислал автор, то он и видит).

### 7.2. Шифрование payload

Содержимое блока может быть зашифровано автором. В этом случае подписывающий видит зашифрованный payload, может его подписать "вслепую" (фактически подписывая хеш). Позже автор может раскрыть ключ для верификации содержимого третьими сторонами.

### 7.3. Двусторонние и односторонние подписи

- **Двусторонние:** обе стороны соглашаются, обмен подписями (используется при слияниях).
- **Односторонние:** одна сторона копирует публично доступный блок другой и подписывает у себя ("я видел это состояние"). Согласие автора не требуется. Автор может узнать об односторонних пломбах при последующих синхронизациях.

### 7.4. Хранение пломб

Пломбы хранятся **у подписывающего**, и опционально (по обмену) у автора. Глобальный реестр отсутствует. Распространение знания о пломбах — через gossip при синхронизациях.

## 8. Приватность

- Ветки **по умолчанию приватны** — известны только автору.
- Ветка становится "публичной" только в результате намеренной публикации, синхронизации с публичным узлом или утечки.
- Все ветки структурно одинаковы. Тип "публичная/приватная" — не свойство ветки, а свойство практики её использования.
- При синхронизации передаётся только путь к нужной ветке (не всё дерево узлов).

---

## 9. Инварианты

Система должна гарантировать соблюдение следующих инвариантов:

1. **Целостность дерева узлов:** для любого некорневого узла `N` существует узел `(N-1)/2`, и подпись `N` валидна по ключу родителя.
2. **Целостность ветки:** для любого блока с индексом `i > 0` в ветке существует блок с индексом `i-1`, и `prev_hash` соответствует.
3. **Валидность подписей:** все подписи в дереве и ветках проходят криптографическую проверку.
4. **Монотонность времени в ветке:** `timestamp_claimed[i] ≥ timestamp_claimed[i-1]` для всех i. (Соблюдение требует доверия к автору.)
5. **Уникальность позиции:** в ветке не может быть двух блоков с одним индексом.
6. **Уникальность узла:** не может быть двух разных узлов с одним heap-индексом в дереве пользователя.
7. **Связь блока и узла:** блок ветки может быть подписан только рабочим ключом листового узла этой ветки (с учётом возможной ротации).

## 10. Хранилище и сериализация

- **База данных:** LMDB (embedded key-value, транзакционная).
- **Структура таблиц:**
  - `nodes`: ключ = `(root_pubkey, node_index)`, значение = сериализованный узел.
  - `blocks`: ключ = `(root_pubkey, node_index, block_index)`, значение = сериализованный блок.
  - `seals`: ключ = хеш блока, значение = список пломб (подписей других пользователей).
  - `external_blocks`: блоки чужих пользователей, скопированные при синхронизации.
- **Сериализация:** CBOR (RFC 8949) с deterministic encoding (раздел 4.2.1) — гарантирует, что один и тот же логический блок всегда сериализуется в идентичные байты.
- **Хеширование:** BLAKE2b-256.
- **Подписи:** Ed25519.
- **Криптобиблиотека:** libsodium.

## 11. Открытые вопросы

Вопросы, которые отложены до более позднего этапа проектирования:

### 11.1. Размещение блоков отзыва (REVOCATION)
В какой ветке размещаются блоки отзыва? В отдельной служебной ветке? В ветке-родителе скомпрометированного узла? Решить при проектировании механизма отзыва.

### 11.2. Защита от создания фальшивых поддеревьев при компрометации
Если злоумышленник получил рабочий ключ узла N, он может создать новые узлы-потомки и ветки. Варианты защиты:
- Свежесозданные узлы становятся "финальными" только после свидетельствования другими пользователями (до этого — отзываемы родителем без сложностей).
- Двойная подпись при создании узла (родитель + прародитель).

Решение: предположительно — комбинация "свидетельствование как условие финализации" + механизм отзыва родителем. Окончательно решить при проектировании протокола синхронизации.

### 11.3. Распространение информации об отзывах
В модели без глобального реестра распространение происходит через gossip. Нужно спроектировать:
- Какие данные передаются о статусе отзывов при синхронизации.
- Как UI отображает "свежесть" знания об отзывах.
- Перепроверка валидности своих пломб при получении новых отзывов.

### 11.4. Отказоустойчивость группового слияния — РЕШЕНО (v0.3)
В DAG-модели (§6.5) оффлайн-участник перестаёт быть проблемой: заглохшая ветвь не разрушает граф, пакет продолжает расти в других слияниях. Синхронный сбор и предварительная договорённость отменены. Остаётся сетевой вопрос обнаружения партнёров — решается через агрегатор (модуль синхронизации).

### 11.5. Восстановление мастер-идентичности
Возможность смены корневого ключа без потери истории (например, через социальное восстановление подписями нескольких доверенных лиц). Не реализуется в MVP, но в структуре корневого узла предусмотреть поле "версия мастер-ключа".

### 11.6. Внешнее хранилище больших данных
Для больших payload (видео, файлы) рассмотреть вынос содержимого во внешнее хранилище (по аналогии с IPFS), с хранением в блоке только хеша и метаданных. На текущем этапе — payload хранится непосредственно в блоке.

### 11.7. Ротация ключей структурного уровня
Механизм ротации рабочего ключа описан (§6.6). Ротация структурного ключа (изменение позиции узла в дереве) — отдельная сложная тема, отложена до перехода на гибридную схему ключей.

### 11.8. Каноничность UTF-8 в payload
Если payload содержит текст, нужна нормализация (например, NFC) для каноничности хеширования. Уточнить при первой реализации.
