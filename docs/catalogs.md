# Универсальные каталоги профессий и потребностей

**Статус:** спецификация справочных данных для теговой конвенции профиля (`records.md` §8.6/§8.7, ИР-005). Решение о хранении (2026-07-13): **гибрид** — версионируемые JSON-файлы в этом каталоге, которые агрегатор раздаёт по `GET /catalog`; позже, при необходимости, записи можно заякорить как `Concept`-ы на референс-цепи проекта.

## Зачем

Чтобы самоописания участников были **машинно-сопоставимы**, навык и потребность выбираются из общего справочника (слаг), а не формулируются каждым по-своему. Живое описание при этом остаётся в `Concept.text` — каталог даёт только устойчивый ключ (`cat:<slug>`), по которому экспорт (§13) стыкует спрос и предложение.

## Файлы

Один файл на каталог, имя = имя каталога:

| Файл | Слаг-префикс | Содержимое |
|------|--------------|------------|
| `catalogs/professions.json` | `prof.` | профессии и навыки |
| `catalogs/needs.json` | `need.` | категории насущных потребностей |
| `catalogs/industries.json` | `ind.` | отрасли для развития |
| `catalogs/hobbies.json` | `hobby.` | хобби / занятия по душе |

## Формат

```json
{
  "catalog": "professions",
  "version": "2026.07",
  "entries": [
    {
      "slug": "prof.electrician",
      "ru": "Электрик",
      "group": "Ремонт и строительство",
      "aliases": ["электромонтёр", "электромонтажник"]
    }
  ]
}
```

Поля записи:

- `slug` — устойчивый идентификатор с префиксом каталога, `[a-z0-9.-]`. **Не переименовывается** после публикации (на него ссылаются профили тегом `cat:<slug>`). Устаревшую запись помечают `"deprecated": true` и по желанию `"superseded_by": "<slug>"`, но не удаляют.
- `ru` — отображаемое имя (русское).
- `group` — раздел для группировки в выборе (человекочитаемый).
- `aliases` — синонимы для поиска (не влияют на слаг).
- `parent` — опциональный слаг родительской записи (иерархия каталога).
- `closed_by` — **только для каталога потребностей:** список слагов профессий, которые эту потребность закрывают (`need.electrical` → `["prof.electrician"]`). Без этого поля стыковка не механизируется: человек понимает, что проводку чинит электрик, а программа — нет. Пустой список означает «профессией не закрывается» (например `need.appliances` — это вещь, а не услуга).

Поля файла:

- `version` — строка `ГГГГ.ММ`. Меняется при любом изменении наполнения; экспорт профилей указывает версию каталога, против которой считан.

## Как участник использует запись

Клиент/писарь выбирает запись и пишет обычный `Concept` (§8.6):

```
Concept{ text:"Электромонтаж, 8 лет, допуск до 1000В",
         tags:["kind:skill","cat:prof.electrician","retrain:no"] }
```

`kind:*` задаёт отношение (умею / хочу освоить / нужно), префикс слага — из какого каталога запись.

## Наполнение потребностей

Базовые потребности у людей во многом совпадают (еда, одежда, жильё, техника, дети, здоровье, отдых) — стартовый `needs.json` покрывает именно этот общий каркас, а дефицит специальностей закрывается переобучением (тег `retrain:*` в профиле, лестница разрядов `records.md` §14.4). Список — семя, расширяется по мере пилота.

## Управление и версионирование

- **Курирование:** правки каталогов идут через изменения в репозитории (PR), как обычные данные проекта — семя намеренно небольшое и растёт от реальных запросов пилота, а не «на всякий случай».
- **Раздача:** агрегатор запускается с `--catalog PATH` (каталог этих файлов) и отдаёт их по `GET /catalog` — связкой `{"professions": {...}, "needs": {...}}` — и по `GET /catalog/<name>` поштучно. Файлы читаются с диска на каждый запрос: правка каталога не требует перезапуска. Без `--catalog` эндпоинты отвечают 501.
- **Валидация слага.** Клиент сверяет каждый тег `cat:` с каталогом и **отказывается писать неизвестный слаг** (`bc concept add`, `bc apply`). Причина строгости: опечатка в слаге не падает громко — человек просто перестаёт находиться по этому навыку. Обойти — `--force` (если каталог ещё не содержит нужного слага; организатор потом дополнит). Если агрегатор недоступен или каталога не отдаёт, валидация не выполняется и писать не мешает.
- **Найти слаг:** `bc catalog --via URL --search ТЕКСТ` — по слагу, названию или синониму.
## Целевое направление: каталог живёт в блоках (2026-07-13)

Файлы в репозитории — стартовый костыль, а не конечная форма. Каталог по природе своей — **та же «любая информация»**, что и профиль, и должен жить в цепях:

- Запись каталога — обычный `Concept` с фасетным тегом `kind:catalog-entry` и слагом (`cat:prof.electrician`), лежащий в чьей-то цепи. Отдельный тип не нужен — ровно как с профилем (§8.6).
- Каталог может вестись в **отдельной «идеологической» цепи** — без трудочасов и экономики, чисто справочные данные.
- **У каждого свой каталог.** Люди ведут свои каталоги в своих цепях; они не обязаны совпадать.
- **Защита от спама — выбором источника, а не модерацией.** Клиент настраивает, какие цепи он признаёт источниками каталога. На старте — «только моя цепь» (или цепь инициатора пилота); дальше круг доверенных источников расширяется по мере надобности. Тот, кто засорил свой каталог мусором, засорил только себя — его просто не выбирают источником.
- Слаг остаётся ключом стыковки (`cat:<slug>`), поэтому каталоги разных источников **совместимы**, пока слаги совпадают: два человека, взявшие `cat:prof.electrician` из разных цепей, всё равно стыкуются.

Переход на этот механизм не ломает уже записанные профили: теги `cat:*` в них не меняются — меняется только то, откуда клиент берёт список для выбора. Поэтому файлы сейчас — безопасный старт.
