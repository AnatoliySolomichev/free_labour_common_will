# Типы записей блокчейна

**Статус:** черновик v0.2
**Основание:** обсуждение семантического слоя, апрель 2026.
**Назначение:** описание структуры данных, хранимых в поле `payload` пользовательских блоков. Является надстройкой над транспортным слоем, описанным в `blockchain.md`.

---

## 1. Назначение

Поле `payload` блока типа `DATA` содержит произвольные байты в формате CBOR. Данный документ определяет стандартные типы записей — семантический слой, на котором строится граф знаний, трудовая экономика и производственные цепочки.

Ключевой принцип: **запись идентифицируется своим содержимым**. Хеш записи (BLAKE2b-256 от её CBOR-представления) является её адресом. Ссылки между записями — контентно-адресованные.

---

## 2. Формат записи

Каждая запись — CBOR-map с обязательным полем `"t"` (тип, `uint8_t`) в качестве первого ключа (по правилам deterministic CBOR ключи сортируются по длине затем лексикографически; `"t"` — 1 байт, сортируется раньше более длинных ключей).

```
Record = CBOR map {
  "t": uint8_t,   // дискриминатор типа (см. таблицу §3)
  ...             // поля, специфичные для типа
}
```

Один блок содержит одну запись или упорядоченный список записей:

```
Block.payload = CBOR encode(Record)
             | CBOR encode([Record, Record, ...])
```

## 3. Дискриминаторы типов

| Диапазон | Категория |
|----------|-----------|
| 0x00–0x0F | Базовые / структурные |
| 0x10–0x1F | Онтологические примитивы |
| 0x20–0x2F | Модификаторы |
| 0x30–0x3F | Программные примитивы |
| 0x40–0x4F | Граф знаний |
| 0x50–0x5F | Трудовые записи |
| 0x60–0x6F | Производство |
| 0x70–0x7F | Экономика |
| 0x80–0xFF | Зарезервировано |

## 3A. Структурные / протокольные записи (0x00–0x0F)

Записи, относящиеся к протоколу блокчейна (`blockchain.md`), а не к семантике знаний/экономики. В отличие от базовых примитивов (§4), имеют собственный дискриминатор.

### 3A.1. FraudClaim (0x01) — Обвинение в структурном обмане

Обвинение merge-блока (`blockchain.md` §6.5) в криптографически невалидном содержимом снимка. Пишется в блокчейн **обвинителя**, ссылается на обвиняемый блок через `Ref` (в чужую цепь писать нельзя). Само обвинение веса не имеет — вес даёт **приложенное самопроверяемое доказательство**; агрегатор/читатель верифицирует его прежде, чем учесть. Это негативный двойник со-подписи слияния (`blockchain.md` §6.5.5): со-подпись = «ручаюсь, что честно», `FraudClaim` = «доказываю, что нет».

**Область действия.** Ловит только **структурный/криптографический** обман. Sybil (валидные, но фальшивые личности) `FraudClaim` не ловит — это задача репутационного слоя (граф экономического обмена, `blockchain.md` §6.5.4).

```
FraudClaim = {
  "t":      0x01,
  "target": Ref,          // обвиняемый merge-блок {chain, hash}
  "kind":   Text,         // "bad_sig" | "hash_mismatch"
  "proof":  Map,          // доказательство, зависит от kind
  "reason": Text?         // человекочитаемое пояснение (опционально)
}
```

Поддерживаются только **позитивно доказуемые** виды. Доказательство отсутствия («блока нет вообще», «count завышен») в открытой сети недоказуемо и в блокчейн не выносится — остаётся локальным игнором.

Оба вида несут общий набор полей (`blockchain.md` §6.5.6): помимо листа и пути Меркла, обязателен **путь узлов** участника — без него нельзя добраться до `working_pubkey` и проверить подпись блока (см. §6.5.5, проблема отсутствия ключа).

```
proof = {
  "leaf":      { "chain": Bytes(32), "node": Int, "block": Int, "hash": Bytes(32) },  // оспариваемый ExternalRef
  "merkle":    { "path": [Bytes(32), ...], "dirs": [Bool, ...] },  // включение H(leaf) в target.merkle_root
  "nodes":     [Node, ...],   // путь узлов root..node цепи chain → working_pubkey листового узла
  "evidence":  Block          // блок-улика (см. ниже, зависит от kind)
}
```

**kind = `"bad_sig"`** — `evidence` = сам блок-0 (его хеш = `leaf.hash`); проверяющий видит, что подпись блока **не проходит** по `working_pubkey`.

**kind = `"hash_mismatch"`** — `evidence` = настоящий подписанный блок участника по адресу `(chain, node, block)`; его подпись валидна, но хеш ≠ `leaf.hash` → зафиксированный хеш подделан.

**Проверка (любой вид):**

1. **Структура.** Пересчитать корень Меркла из `H(leaf)` и `merkle`, сверить с `target.merkle_root`; проверить путь узлов `nodes` (корень самоподписан и `= chain`, каждый узел подписан родителем, последний = `leaf.node`). Любой провал → доказательство **сфабриковано**. Из последнего узла берётся `working_pubkey`.
2. **Дефект.** `bad_sig`: `evidence` — тот же блок (`hash = leaf.hash`) и его подпись невалидна по `working_pubkey`. `hash_mismatch`: `evidence` по тому же адресу, подпись валидна, а хеш ≠ `leaf.hash`.

**Исход и вес:**

| Исход                   | Условие                                          | Вес                               |
|-------------------------|--------------------------------------------------|-----------------------------------|
| CONFIRMED               | п.1 и п.2 сошлись                                | сильный минус обвиняемому         |
| REFUTED (добросовестно) | путь сходится к корню, но дефект ложный          | ноль; штрафа обвинителю нет       |
| REFUTED (фабрикация)    | путь **не** восстанавливает `target.merkle_root` | ноль + штраф репутации обвинителю |

Штраф обвинителю — **только** за фабрикацию (провал п.1): это провабельно злонамеренно. Ошибка в дефекте при верном пути (п.1 прошёл, п.2 нет) штрафом не наказывается — обвинитель ткнул в реальный лист, но добросовестно ошибся.

**Проверка vs построение.** Само поле `proof` — непрозрачный blob (сериализованный `FraudProofData` транспортного слоя); `records` его не интерпретирует. *Проверка* реализована и самодостаточна: `blockchain::FraudProof::verify(kind, proof, merkle_root_of(target))` (CLI: `bc fraud verify`). *Построение* же требует сырого набора листьев обвиняемого пакета, путей узлов и блоков-улик — данных кеш/gossip-слоя, отложенных до модуля синхронизации (`blockchain.md` §6.5.6, §11.9).

## 4. Базовые типы (примитивы данных)

Используются как значения полей в других записях. Самостоятельного дискриминатора не имеют — это нативные типы CBOR.

| Тип | CBOR | Описание |
|-----|------|----------|
| `Null` | `0xf6` | отсутствие значения |
| `Bool` | `true`/`false` | логическое значение |
| `Int` | int / uint | целое (до 64 бит) |
| `Float` | float64 | вещественное |
| `Text` | text string | UTF-8, NFC-нормализация обязательна |
| `Bytes` | byte string | сырые байты |
| `List` | array | упорядоченная последовательность |
| `Map` | map | именованные поля {key: value} |

### Ссылка (Ref)

Ссылка на запись в любом блокчейне — основной механизм связей между записями и пользователями.

```
Ref = {
  "chain": Bytes(32),   // UserId: публичный ключ корневого узла владельца
  "hash":  Bytes(32)    // BLAKE2b-256 от CBOR-сериализации referenced блока
}
```

Верификация ссылки: загрузить блок с данным `hash` от пользователя `chain`, проверить совпадение хеша. Возможна как P2P (напрямую к владельцу), так и через агрегатор (см. §13).

### Quantity (встроенный)

Используется как значение поля, не как самостоятельная запись.

```
Quantity = { "v": Float, "u": Text }   // v: значение, u: единица ("час", "кг", "м³", ...)
```

## 5. Онтологические примитивы (0x10–0x1F)

Базовые категории для описания реальности. Из этих примитивов строятся все остальные типы.

### 5.1. Entity (0x10) — Сущность

Поименованная вещь или субъект.

```
Entity = {
  "t":    0x10,
  "name": Text,
  "desc": Text?,
  "attr": Map?           // произвольные атрибуты {key: value}
}
```

### 5.2. Process (0x11) — Процесс

Изменение во времени.

```
Process = {
  "t":    0x11,
  "name": Text,
  "desc": Text?
}
```

### 5.3. State (0x12) — Состояние

Снимок сущности в момент времени.

```
State = {
  "t":      0x12,
  "entity": Ref | Text,
  "value":  any,
  "at":     Int?          // Unix timestamp (опционально)
}
```

### 5.4. Event (0x13) — Событие

Процесс в конкретный момент и место.

```
Event = {
  "t":       0x13,
  "process": Ref | Text,
  "agent":   Ref?,
  "at":      Int?,
  "loc":     Ref | Text?,
  "inputs":  [Ref | Quantity]?,
  "outputs": [Ref | Quantity]?
}
```

### 5.5. Property (0x14) — Свойство

Характеристика сущности.

```
Property = {
  "t":     0x14,
  "name":  Text,
  "value": any,
  "unit":  Text?
}
```

### 5.6. Relation (0x15) — Отношение

Направленная связь между двумя сущностями.

```
Relation = {
  "t":    0x15,
  "from": Ref,
  "to":   Ref,
  "kind": Text    // "является частью", "зависит от", "применяет", "порождает", ...
}
```

### 5.7. TimeRecord (0x16) — Время

```
TimeRecord = {
  "t":    0x16,
  "kind": "moment" | "period",
  "from": Int,             // Unix timestamp, UTC
  "to":   Int?             // только для "period"
}
```

### 5.8. Location (0x17) — Место

```
Location = {
  "t":    0x17,
  "name": Text?,
  "geo":  [Float, Float]?,  // [lat, lon]
  "ref":  Ref?              // ссылка на внешнюю запись места
}
```

### 5.9. Role (0x18) — Роль

Сущность в конкретном контексте (например: человек в роли электрика 3-го разряда).

```
Role = {
  "t":      0x18,
  "entity": Ref,            // ссылка на Worker или другую Entity
  "as":     Ref | Text,     // специальность, Grade или должность
  "in":     Ref?            // кооператив/организация (опционально)
}
```

## 6. Модификаторы (0x20–0x2F)

Изменяют смысл другой записи не создавая новую сущность.

### 6.1. Negation (0x20) — Отрицание

```
Negation = { "t": 0x20, "of": Ref }
```

### 6.2. Degree (0x21) — Степень

```
Degree = { "t": 0x21, "of": Ref, "k": Float }   // k: коэффициент 0.0–∞
```

### 6.3. ConditionMod (0x22) — Условный модификатор

Применяет запись только при выполнении условия.

```
ConditionMod = {
  "t":    0x22,
  "when": Ref,     // предикат (ссылка на запись-условие)
  "then": Ref      // применяемая запись
}
```

### 6.4. Approximation (0x23) — Приближение

```
Approximation = { "t": 0x23, "of": Ref, "from": any, "to": any }
```

### 6.5. TemporalMod (0x24) — Временной модификатор

```
TemporalMod = {
  "t":    0x24,
  "of":   Ref,
  "kind": "always" | "sometimes" | "during" | "never",
  "time": Ref?
}
```

### 6.6. CausalLink (0x25) — Причинно-следственная связь

```
CausalLink = {
  "t":      0x25,
  "cause":  Ref,
  "effect": Ref
}
```

## 7. Программные примитивы (0x30–0x3F)

Позволяют описывать алгоритмы и правила как данные (гомоиконичность). Используются для формализации правил расчёта коэффициентов, валидации записей и описания алгоритмов в базе знаний.

### 7.1. Variable (0x30)

```
Variable = { "t": 0x30, "name": Text, "type": Text? }
```

### 7.2. Binding (0x31)

```
Binding = { "t": 0x31, "var": Text, "val": any }
```

### 7.3. IfThen (0x32)

```
IfThen = {
  "t":    0x32,
  "cond": any,       // предикат
  "then": any,       // ветка «истина»
  "else": any?       // ветка «ложь» (опционально)
}
```

### 7.4. ForEach (0x33)

```
ForEach = {
  "t":    0x33,
  "var":  Text,
  "in":   Ref | List,
  "body": any
}
```

### 7.5. While (0x34)

```
While = {
  "t":    0x34,
  "cond": any,
  "body": any
}
```

### 7.6. Sequence (0x35)

```
Sequence = {
  "t":     0x35,
  "steps": [any, ...]
}
```

### 7.7. Function (0x36)

```
Function = {
  "t":      0x36,
  "name":   Text,
  "params": [Text, ...],    // имена параметров
  "body":   any
}
```

### 7.8. Apply (0x37)

```
Apply = {
  "t":    0x37,
  "fn":   Ref,
  "args": [any, ...]
}
```

### 7.9. BinaryOp (0x38)

```
BinaryOp = {
  "t":  0x38,
  "op": "+" | "-" | "*" | "/" | ">" | "<" | "=" | "!=" | "and" | "or",
  "l":  any,
  "r":  any
}
```

### 7.10. UnaryOp (0x39)

```
UnaryOp = {
  "t":  0x39,
  "op": "not" | "neg",
  "v":  any
}
```

**Пример — правило валидации приёмки:**

```
rule_accept = IfThen {
  cond: BinaryOp { op: "=", l: Variable "quality", r: "пройдено" },
  then: Apply { fn: Ref(fn_compute_labor_units), args: [Variable "hours", Variable "grade"] },
  else: Binding { var: "labor_units", val: 0 }
}
```

## 8. Граф знаний (0x40–0x4F)

### 8.1. Concept (0x40) — Идея / концепция

Атомарная единица знания.

```
Concept = {
  "t":    0x40,
  "text": Text,
  "tags": [Text, ...]?
}
```

### 8.2. ConceptLink (0x41) — Связь между концепциями

```
ConceptLink = {
  "t":    0x41,
  "from": Ref,
  "to":   Ref,
  "kind": Text    // "уточняет", "противоречит", "применяет", "порождает", ...
}
```

### 8.3. Composite (0x42) — Обобщающая идея

Группа связанных концепций, объединённых под общим заголовком.

```
Composite = {
  "t":     0x42,
  "title": Text,
  "parts": [Ref, ...]    // ссылки на Concept, ConceptLink, другие Composite
}
```

Хеш Composite является его идентификатором. Два Composite с одинаковым составом и порядком частей имеют одинаковый хеш — это их «тип» по содержанию.

**Открытый вопрос 14.3:** порядок в `parts` влияет на хеш. Нужно решить: сортировать ли ссылки (детерминированный хеш) или сохранять авторский порядок (семантически значимый).

### 8.4. Copy (0x43) — Копия чужой записи

Механизм реакций: пользователь не может писать в чужой блокчейн. Чтобы выразить отношение к чужой записи — нужно скопировать её к себе.

```
Copy = {
  "t":      0x43,
  "source": Ref    // ссылка на оригинальную запись в чужом блокчейне
}
```

### 8.5. Reaction (0x44) — Реакция

Выражение отношения к записи. Реакция всегда указывает на `Copy` (чужая запись) или на собственную запись.

```
Reaction = {
  "t":      0x44,
  "target": Ref,     // ссылка на Copy или на свою запись
  "value":  Int      // int8_t: -128..+127
}
```

**Семантика значений:**

| Диапазон | Смысл |
|----------|-------|
| -128..-64 | Сильное несогласие / ошибочность |
| -63..-1 | Частичное несогласие |
| 0 | Нейтрально / зафиксировано без оценки |
| +1..+63 | Частичное согласие / полезно |
| +64..+127 | Сильное согласие / важно |

**Правило одной реакции:** на один `target` — одна активная `Reaction`. При изменении оценки добавляется новый блок с новой `Reaction` на тот же `target`; агрегаторы учитывают только последнюю (по порядковому индексу блока в ветке, не по `timestamp_claimed`).

### 8.6. Профиль и потребности — теговая конвенция над Concept (решает §14.6)

Самоописание участника (навыки, потребности, желания) выражается **не отдельным протокольным типом, а обычными `Concept`-ами** с зарезервированными тегами. Мотив: профиль — это «любая информация», он должен жить на уровне идей, а не протокола (§14.6). Экономические `Specialty`/`Worker`/`Grade` (§9) — отдельный слой про расчётные ставки; профиль-концепты — про «кто есть / кому что нужно / кто чего хочет», их можно связать через `ConceptLink`.

**Модель (решение 2026-07-13): атомы + опциональная карточка.** Каждый факт профиля — отдельный `Concept`-блок (легко дополнять и обновлять по одному, на каждый можно `Copy`+`React`). По желанию участник группирует их в `Composite` «карточку профиля» — аккуратную единую точку входа; экспорт (§13) всё равно сканирует по тегам и не требует карточки.

**Фасет — обязателен.** Ровно он делает `Concept` фактом профиля; без него это обычная идея, и экспорт её не видит.

| Фасет | Значение |
|-------|----------|
| `kind:skill` | навык / профессия, которыми владею |
| `kind:need` | насущная потребность / запрос |
| `kind:aspiration` | профессию хочу освоить |
| `kind:industry` | отрасль хочу развивать |
| `kind:hobby` | хобби / что люблю делать |
| `kind:obstacle` | что мешает закрыть потребности |

`kind:` задаёт **отношение** участника к записи каталога, а префикс слага (`prof.`/`need.`/`ind.`/`hobby.`) — из какого каталога она взята: одна запись `cat:prof.welder` под `kind:skill` значит «умею варить», под `kind:aspiration` — «хочу научиться». `Concept.text` несёт живое человеческое описание.

**Базовые характеристики (принято 2026-07-13).** Минимум, достаточный для осмысленной стыковки. Профиль, который долго заполнять, не заполнят — поэтому набор намеренно узок; всё остальное добавляется потом, бесплатно, новыми тегами.

| Тег | Значение |
|-----|----------|
| `cat:<slug>` | запись каталога (§8.7), напр. `cat:prof.electrician` |
| `geo:<lat>,<lon>` | где это актуально. Проект всепланетарный: ремонт машины важен в конкретном месте |
| `r:<km>` \| `r:global` | радиус актуальности. Он же — ручка приватности: больше радиус = обезличеннее |
| `remote:yes` \| `no` | можно ли удалённо (консультация, онлайн-работа → часто `r:global`) |
| `horizon:now` \| `3m` \| `1y` \| `3y` \| `5y` | горизонт потребности/желания |
| `retrain:yes` \| `no` \| `maybe` | готовность осваивать новую профессию |
| `grade:1..6` | разряд — мост к экономическому `Grade` (§9.2) |
| `exp:<лет>` | стаж по навыку |
| `capacity:<часов/нед>` | сколько может отдать труду |
| `urgency:low` \| `normal` \| `high` | срочность потребности |

**Приватность через огрубление.** Точность координаты выбирает сам участник: `geo:55.7` — район, `geo:55.7512` — дом. Кто хочет обезличиться — огрубляет координату и увеличивает радиус.

**Резерв на будущее** (не обязательны, вводятся по мере надобности — код их уже пропускает насквозь, см. §13): `mobile:yes|no` (готов выезжать) · `hours:09:00-18:00` · `days:mon-fri|weekends` · `lead:<дней>` (за сколько предупредить) · `duration:<часов>` (объём работы) · `deadline:<unix>` · `season:summer|winter` · `cert:<название>` (допуск, напр. «до 1000В») · `tools:yes` (со своим инструментом) · `team:<n>` (может привести бригаду) · `teach:yes` (готов обучать этому) · `barter:ok` (встречный обмен навыком) · `free:yes` (безвозмездно) · `price:<трудочасов>` (ожидаемая оценка) · `lang:ru,en` · `for:children|elderly` · `mind:tech|humanities` (склад ума).

```
Concept{ text:"Электромонтаж, 8 лет, допуск до 1000В",
         tags:["kind:skill","cat:prof.electrician","geo:55.75,37.62","r:30",
               "grade:4","exp:8","capacity:20","retrain:maybe"] }
Concept{ text:"Нужно заменить проводку в двушке, ищу электрика",
         tags:["kind:need","cat:need.electrical","geo:55.76,37.60","r:10",
               "horizon:now","urgency:high"] }
Concept{ text:"Консультирую по бухгалтерии, где угодно",
         tags:["kind:skill","cat:prof.accountant","remote:yes","r:global"] }
```

**Жизненный цикл потребности — словарь сделки (решение 2026-07-15, ИР-006 шаг 2).** Весь путь от найденного матча до закрытой потребности выражается обычными `ConceptLink` — ни одного нового протокольного типа:

| Связь | from → to | Чья цепь | Смысл |
|-------|-----------|----------|-------|
| `kind:"берусь"` | свой навык → чужая потребность | исполнителя | заявка на исполнение (решает §14.7) |
| `kind:"исполняет"` | своя `WorkRecord` → потребность | исполнителя | эта работа сделана под эту потребность; **много-ко-многим** — несколькими связями (три захода на одну потребность; один заказ на две) |
| `kind:"закрыто"` | своя `Acceptance` → своя потребность | заказчика | авторитетное снятие: заодно фиксирует, **чем** закрыто |

**Все связи необязательны.** Сделка — не жёсткий конвейер, а потребность с хвостом ссылок на неё: прямая сделка без постнутой потребности, работа на склад без покупателя, закрытие без работы в системе (помог сосед) — законные состояния, а не ошибки. Обещание (`Pledge`) — тоже необязательное звено: оно про финансирование наперёд (§11.3), а не про личность сделки. Стадия сделки выводится из наличных звеньев.

**Закрыть факт может только его хозяин:** закрывающая связь обязана лежать в той же цепи, на которую указывает (`to.chain` == автор блока) — иначе чужой мог бы «снять» вашу потребность. Если закрывать нечем (работа вне системы), допускается прежнее самозамыкание `from == to` (потребность → она же). Закрытый факт помечается в экспорте (`"closed": true`) и выпадает из стыковки.

### 8.7. Универсальные каталоги профессий и потребностей

Курируемые справочники, из которых участник выбирает запись (`cat:<slug>`), а не изобретает формулировку — так самоописания машинно-сопоставимы. Хранение (решение 2026-07-13): **гибрид** — версионируемые JSON-файлы в репозитории (`docs/catalogs/`), которые агрегатор раздаёт по `GET /catalog`; позже, при необходимости, каталог можно заякорить как `Concept`-ы на референс-цепи проекта. Формат и наполнение — в `docs/catalogs.md`.

### 8.8. Черновик: готовит писарь, подписывает хозяин (решение 2026-07-13)

Нетехническому участнику (электрику, швее, повару) сырой CLI недоступен. Поэтому записи за него может подготовить **помощник — писарь, бухгалтер или ИИ**. Разрез проходит по границе `payload`: содержимое записи не требует ни ключей, ни состояния цепи; подпись требует и того, и другого.

**Черновик** — key-free JSON со списком записей. Он **не несёт полномочий**: его можно слать любым каналом и генерировать коммерческой моделью. Ценность появляется только когда хозяин применяет его своим ключом — `bc apply --draft FILE`, где **один блок = одна подпись** (`blockchain.md` §5).

```json
{ "leaf": 2147483647,
  "records": [
    {"t":"concept","text":"Пошив и ремонт одежды, 12 лет",
     "tags":["kind:skill","cat:prof.tailor","geo:55.75,37.61","r:15","grade:4"]}
  ] }
```

**Белый список — применение «градиента необратимости» (ИР-002).** В черновике допустимы только полностью обратимые записи без ценности: `Concept`, `ConceptLink`, `Composite`. Ценностные и необратимые (`Transfer`, `Pledge`, `Acceptance`, `Redemption`, операции с ключами) **отвергаются на входе** — не бдительностью читателя, а структурно. Поэтому перевод не может спрятаться среди тридцати фактов профиля. Обоснование асимметрии: ошибочно подписанный `Concept` снимается следующей записью (§8.6), а ошибочно подписанный `Transfer` необратимо двигает ценность.

**Почему пакет, а не поштучное подтверждение.** Каждый блок и так несёт свою подпись — «подписать много блоков одной подписью» невозможно физически. Вопрос лишь в человеческом просмотре, и здесь пятнадцать подтверждений подряд **учат жать «да» не читая**: усталость от подтверждений — сама по себе уязвимость. Поэтому: потолок в **20 записей** на черновик, **полный рендер каждой** записи без обрезки (`--dry-run`), одно осознанное подтверждение. Отклонение черновика — всегда целиком, включая безобидные записи.

Ценностные записи, когда их допустят в черновики, подписываются **по одной**, с ручным переподтверждением суммы и получателя; смешанный черновик запрещён.

## 9. Трудовые записи (0x50–0x5F)

### 9.1. Specialty (0x50) — Специальность

```
Specialty = {
  "t":    0x50,
  "name": Text     // слаг каталога: "prof.electrician", "prof.carpenter", ...
}
```

**`name` — это слаг каталога (§8.7), а не вольное название (решение 2026-07-13, ИР-006).** Причина жёсткая: `name` служит **глобальным ключом ставок** по всей сети (§11.2 считает ставку по паре «имя специальности + разряд»). Пока это был свободный текст, «Электрик», «электрик» и «Электромонтёр» давали **три разные ставки** и рвали статистику на куски. Слаг из общего каталога — готовый единый ключ, уже используемый профилями (`cat:prof.electrician`), поэтому одним решением чинится и статистика, и разрыв между профилем и экономикой: человек и его труд говорят на одном ключе.

Клиент проверяет слаг по каталогу и **отказывается заводить неизвестный** — как и при записи профиля (§8.7). Человеческое название берётся из каталога (`ru`) и в цепь не пишется: оно может меняться, ключ — нет.

### 9.2. Grade (0x51) — Разряд специальности

```
Grade = {
  "t":         0x51,
  "specialty": Ref,    // ссылка на Specialty
  "level":     Int     // 1–6
}
```

### 9.3. Worker (0x52) — Труженик

```
Worker = {

  "chain":  Bytes(32),    // UserId владельца (совпадает с корневым ключом его блокчейна)
  "grades": [Ref, ...]?   // ссылки на Grade (подтверждённые специальности)

### 9.4. WorkRecord (0x53) — Запись о труде

Один акт труда. Трудочасы **не возникают** при создании записи — только после `Acceptance`.

```
WorkRecord = {
  "t":      0x53,
  "agent":  Ref,         // ссылка на Role (Worker + Grade в контексте)
  "action": Text,        // описание работы
  "time": {
    "start": Int,        // Unix timestamp начала
    "hours": Float       // затраченное время в часах
  },
  "inputs": [{
    "resource": Ref | Text,
    "qty":      Float,
    "unit":     Text
  }, ...]?,
  "outputs": [{
    "resource": Ref | Text,
    "qty":      Float,
    "unit":     Text
  }, ...]?,
  "carry": [{              // v2 (ИР-011): перенос стоимости средств производства
    "src":     Ref,        //   Tool (§10.2) или партия Material (§10.1)
    "used":    Float,      //   израсходовано: часы работы инструмента / количество материала
    "carried": Float,      //   перенесено в эту работу, трудочасов
    "seq":     Int,        //   звено нити переноса записи src
    "prev":    Ref?,       //   предыдущее звено (у первого отсутствует)
    "after":   Float       //   собрано по записи src после звена; инвариант: after ≤ cost
  }, ...]?
}
```

**Нить переноса (v2, ИР-011).** Стоимость инструмента и материалов переносится в продукцию той же механикой, что нить эмиссии (`economy.md` §4.3): каждое использование — звено `{seq, prev, after}`; нить **одна на запись средства производства на всю цепь** — звенья могут писаться в любых ветках (`prev` — Ref, работает через ветки), но линейность обязательна: два звена с одним `prev` — эквивокация, двойное списание одного средства по параллельным веткам ловится как форк нити. Правила признания звена:

- `carried = min(used / ёмкость × cost, cost − собрано_до)` — доля **недособранной** стоимости; ёмкость для инструмента — ресурс `life` в часах работы, для партии материала — её размер `qty`;
- `after = собрано_до + carried`, и всегда `after ≤ cost` — за всю жизнь записи переносится не больше вложенного (аудируемый инвариант, проверяется по последнему звену за O(1));
- при `собрано_до = cost` перенос **обнуляется принудительно**: использование пишется дальше (часы и расход — для статистики и износа), но `carried = 0` — полностью амортизированное средство удешевляет продукцию автоматически; продолжение переноса было бы денежным насосом (§4.2 `economy.md`).

Часы работы инструмента (`used`) заявляет владелец в подписанном слоте — как и часы собственного труда; завышение видно заказчику в разбивке приёмки (решение бинарное), занижение — подарок заказчику за свой счёт.

### 9.5. Acceptance (0x54) — Приёмка

Сертификат качества и **оценка стоимости** работы. Создаётся получателем в своём блокчейне. Сама приёмка трудочасов не создаёт (§12.2) — она фиксирует, сколько работа стоит (`labor_units`); оплата производится `Transfer`'ом (§11.1) со ссылкой на приёмку.

```
Acceptance = {
  "t":           0x54,
  "work":        Ref,         // ссылка на WorkRecord или ProductionChain
  "receiver":    Bytes(32),   // UserId получателя (= владелец этой записи)
  "quality":     Text,        // результат проверки: "пройдено", "отклонено", ...
  "hours_raw":   Float,       // фактические часы из WorkRecord
  "labor_units": Float,       // hours_raw * coefficient(Grade) в день приёмки
  "timestamp":   Int,         // Unix timestamp
  "carried_units": Float?     // v2 (ИР-011): перенесённая стоимость = Σ carried
}
```

Если продукция не дошла до получателя или не прошла проверку — `Acceptance` не создаётся. Стоимость такой продукции равна нулю.

**Перенесённая стоимость (v2, ИР-011).** Если принятый `WorkRecord` несёт `carry` (§9.4), приёмка фиксирует и вторую составляющую цены: `carried_units` = Σ `carried` по звеньям переноса (расхождение — приёмка не признаётся; тот же паттерн «хранится явно, проверяется на равенство», что `labor_units`). **Потолок оплаты** (§12.8 «=») — `labor_units + carried_units`. В ставки §11.2 входит **только** `labor_units`: перенесённая стоимость — овеществлённый труд прошлых периодов и не должна искажать цену живого труда (тот же принцип, что скидки на б/у, `economy.md` §5б). Строгая эквивалентность соблюдена: перенос — это когда-то *принятый* труд (покупка `paid` заверена приёмкой; декларация `est` помечена и дисциплинируется бинарным отказом заказчика).

### 9.6. NegativeWorkRecord (0x55) — Отрицательный труд

Намеренное уничтожение или порча уже принятой продукции.

```
NegativeWorkRecord = {
  "t":          0x55,
  "acceptance": Ref,     // что было уничтожено
  "reason":     Text,

```

### 9.7. Coefficient (0x56) — Локальная подстройка ставки

Переназначен (2026-07-08): суточные ставки публикует `DailyAggregate` (§11.2) в цепи агрегатора. `Coefficient` — **локальная подстройка кооператива/предприятия** поверх сетевой ставки («у нас пекари ×1,15 — дефицит»), федеративный цикл `main_ideas.pdf`: сдавать подстройки никуда не нужно — сделки по подстроенным ценам сами входят во всеобщее усреднение. Не реализуется, пока нет кооперативов (0x64).

## 10. Производство (0x60–0x6F)

### 10.1. Material (0x60) — Партия материала

v2 (ИР-011): материал учитывается **партиями** с переносимым остатком — купленная партия переносит стоимость в продукцию по мере расхода той же нитью переноса, что инструмент (§9.4). Раскладка полей зеркальна `Tool` (§10.2); ёмкость нити — размер партии `qty`.

```
Material = {
  "t":      0x60,
  "name":   Text,
  "unit":   Text,     // "кг", "м³", "шт", "л", "кВт·ч", ...
  "desc":   Text,     // может быть пустым
  "cost":   Float,    // стоимость партии, трудочасов
  "qty":    Float,    // размер партии в unit (ёмкость нити переноса)
  "basis":  Text,     // "paid" | "est"
  "src":    Ref?,     // paid: приёмка покупки — стоимость заверена
  "origin": Ref?,     // предыдущая запись партии (перевыпуск, правила §10.2)
  "note":   Text      // est: пояснение оценки; может быть пустым
}
```

Природные ресурсы (руда, вода, земля) и энергия **не имеют трудочасовой стоимости сами по себе**. Стоимость появляется только через `WorkRecord` на их добычу или генерацию — либо, в переходный период, через декларированный фиатный эквивалент купленной партии (`est`: цена в валюте / средняя часовая ставка, с пометкой всюду). Мелочь (крепёж, расходники на копейки) в партии не оформляется — сидит в k, как накладные расходы мастера.

### 10.2. Tool (0x61) — Инструмент / оборудование

v2 (ИР-011): инструмент несёт стоимость и расчётный ресурс; износ переносит стоимость в продукцию через `carry` (§9.4).

```
Tool = {
  "t":      0x61,
  "name":   Text,
  "desc":   Text,     // может быть пустым
  "serial": Text,     // серийник экземпляра (связь между цепями); может быть пустым
  "cost":   Float,    // стоимость приобретения, трудочасов
  "life":   Float,    // расчётный ресурс, часов работы (ёмкость нити переноса)
  "basis":  Text,     // "paid" | "est"
  "src":    Ref?,     // paid: приёмка покупки — стоимость заверена и проверяема
  "origin": Ref?,     // предыдущая запись экземпляра (перевыпуск)
  "note":   Text      // est: пояснение оценки; может быть пустым
}
```

**Два пути ввода.** `paid` — куплен внутри системы: `src` → `Acceptance` покупки, `cost` обязан совпадать с её стоимостью (проверяемо по ссылке). `est` — существующий инструмент вводится декларацией владельца: стартовая оценка «цена в валюте / средняя часовая ставка» (метод ИР-011), всюду показывается с пометкой «оценка владельца»; дисциплинируется бинарным согласием покупателя продукции — как с эмитентами порций (§12.7).

**Перевыпуск (`origin`) — одна механика на три случая.** Новая запись того же экземпляра со ссылкой на предыдущую; правило признания: **`cost` новой ≤ остаток предыдущей** (её `cost` − `after` последнего звена нити). Случаи: (1) **перепродажа** — перевыпуск в цепи покупателя (`paid`, `src` → приёмка покупки; потолок цены = остаток продавца — учётная остаточная стоимость, `economy.md` §5б); (2) **переоценка** — перевыпуск в своей цепи, **только вниз**: признание внесистемного износа — самонаказание (владелец режет собственный несобранный остаток), поэтому доверия не требует; вверх — денежный насос, не признаётся; (3) **повторный ввод** после перерыва. Цепочка `origin` — прослеживаемая история экземпляра (вместе с `serial` — между цепями); глобальный инвариант «Σ переноса за жизнь экземпляра ≤ стоимость первого ввода» следует из локальных нитей и правила перевыпуска по индукции. Расчётный ресурс `life` заявляет владелец; ошибся — риск его: ранняя смерть инструмента съедает несобранный остаток, долгожительство даёт бесплатные машино-часы (симметрия рисков, ИР-011).

### 10.3. ProductionStep (0x62) — Производственный шаг

Одно преобразование: входные ресурсы → труд → выходные ресурсы.

**Заметка v2 (ИР-011):** учёт входов/выходов и переноса стоимости живёт в самом `WorkRecord` (§9.4 `inputs`/`outputs`/`carry`) — весь конвейер приёмки и оплаты смотрит туда. `ProductionStep` оставлен для будущих **многошаговых цепочек** (`ProductionChain`, §10.4); его формат будет пересмотрен при их реализации — возможное дублирование полей с `WorkRecord` признано и отложено.

```
ProductionStep = {
  "t":       0x62,
  "work":    Ref,        // ссылка на WorkRecord
  "loc":     Ref?,       // место (цех, узел сети)
  "inputs":  [{ "resource": Ref, "qty": Float }, ...]?,
  "outputs": [{ "resource": Ref | Text, "qty": Float }, ...]?
}
```

### 10.4. ProductionChain (0x63) — Производственная цепочка

Упорядоченная последовательность шагов: сырьё → промежуточные детали → перевозка → финальный продукт.

```
ProductionChain = {
  "t":     0x63,
  "name":  Text,
  "steps": [Ref, ...]    // упорядоченные ссылки на ProductionStep
}
```

### 10.5. Cooperative (0x64) — Кооператив

```
Cooperative = {
  "t":       0x64,
  "name":    Text,
  "members": [Bytes(32), ...],    // UserId участников

  "desc":    Text?

## 11. Экономика (0x70–0x7F)

### 11.1. Transfer (0x70) — Передача трудочасов

Единственный способ обмена ценностью в системе. Часы **именные** (§12.7): каждая порция несёт эмитента — цепь, чей долг её обеспечивает. **Кошельки — по веткам** (`economy.md` §5а, v0.6): тратящая ветка = ветка, в которую записан блок перевода (её ключ и подписывает трату); получателю кошелёк указывает поле `to_node`. Долг остаётся на уровне цепи — должен человек, из какого бы кармана ни тратил.

```
Transfer = {
  "t":         0x70,
  "from":      Bytes(32),     // UserId отправителя
  "to":        Bytes(32),     // UserId получателя
  "origins":   [{             // именные порции; сумма units = сумма перевода
      "issuer": Bytes(32),    //   эмитент (цепь-должник)
      "units":  Float         //   количество его часов
  }, ...],
  "reason":    Ref?,          // ЗА ЧТО плачу: Acceptance (обязателен, §12.9)
  "settles":   Ref?,          // КАКОЕ ОБЕЩАНИЕ закрываю: Pledge (v4, опционально)
  "timestamp": Int
}
```

- **Самоэмиссия** (`issuer == from`): рождение новой пары «долг/требование» — плательщик уходит в минус по собственной бумаге (§12.2). Никакого предварительного баланса не требуется.
- **Погашение** (`issuer == to`): бумага вернулась к эмитенту — его долг уменьшается, порция аннигилирует.
- **Передача чужой бумаги**: обычное обращение (индоссамент). Получатель волен отказаться от сделки, если эмитенты порций ему не внушают доверия, — курсов и дисконтов нет, решение бинарное (§12.7).

**Transfer v4 — `settles` (решение 2026-07-13, ИР-006).** У перевода было **одно** поле основания, а нужно оно двум разным вопросам: *за что плачу* и *какое обещание этим закрываю*. Погашение `Pledge` засчитывалось, когда `reason` указывал на обещание, — но §12.9 требует, чтобы `reason` указывал на **приёмку**, иначе перевод не признаётся. Одновременно одно поле это не выражало, и обещание, честно оплаченное трудом, навсегда оставалось «активным» и протухало.

Поэтому вопросы разведены по разным полям:

- `reason` → **`Acceptance`**: за что плачу. Обязателен, как и был (§12.9).
- `settles` → **`Pledge`**: какое обещание этим закрывается. Опционален — прямая сделка без обещания его не несёт.

Погашение обещания считается **только** по `settles`. Прежний путь (`reason` → `Pledge`) отменён: он противоречил §12.9 и был мёртвым.

**Transfer v3 — реализовано (2026-07-11):** `reason` обязателен — перевод без основания не признаётся и отклоняется при приёме (строгая эквивалентность, `economy.md` §4.2, §12.9); выплаты не превышают `labor_units` приёмки, а в статистику ставок входят только **точно** рассчитанные сделки (§12.8 «=», §11.2). Самоэмитирующий перевод несёт звено **нити эмиссии** (`economy.md` §4.3) — опциональную группу полей:

```
  "emission": {               // присутствует ⇔ есть порция issuer == from
    "seq":        Int,        //   сквозной счётчик звеньев цепи
    "prev":       Ref?,       //   предыдущее звено (у первого отсутствует)
    "debt_after": Float       //   декларируемый долг после операции
  }
```

Самоэмиссия без звена нити отклоняется при приёме. Погашение нанизывается на ту же нить квитанцией `Redemption` (§11.5). CLI ведёт нить автоматически (`bc pay`, `bc transfer send/recv`, `bc wallet`).

### 11.2. DailyAggregate (0x71) — Суточные ставки специальностей

Запись в **собственной цепи агрегатора** (автор блока и есть агрегатор — отдельное поле не нужно; подпись, история и перепроверяемость бесплатны). Клиент при нескольких агрегаторах берёт медиану (устойчива к одиночному вранью); действует последняя запись за дату от данного агрегатора (§14.5).

Ключ ставки — глобальная пара (имя специальности, разряд): `Grade`-записи персональны для цепей, статистике нужен межцепной ключ.

```
DailyAggregate = {
  "t":    0x71,
  "date": Int,                     // начало суток UTC (ts − ts mod 86400)
  "rates": [{
    "specialty": Text,             // имя специальности (глобальный ключ)
    "level":     Int,              // разряд 1–6
    "rate":      Float,            // стч/час, после сглаживания
    "hours":     Float,            // Σ hours_raw вошедших сделок дня
    "deals":     Int               // число вошедших сделок
  }, ...],
  "timestamp": Int
}
```

Семантика вычисления (`economy.md` §8.2): агрегат с датой D считается по сделкам **предыдущих суток** (D−86400…D) — «коэффициенты этого дня — из сделок вчерашнего»; входят только **рассчитанные** сделки (приёмка + оплата по ней, самосделки исключены); `среднее_дня = Σ labor_units / Σ hours_raw` по (специальность, разряд); сглаживание `rate = α·среднее_дня + (1−α)·вчерашний rate` (α = 0.3); при объёме дня ниже порога ставка наследуется. Разрешение (специальность, разряд) сделки — по цепочке ссылок `Acceptance → WorkRecord → Grade → Specialty`; неразрешимые сделки не входят.

**v2 (ИР-011):** «рассчитанная» сделка — оплата ровно `labor_units + carried_units` (§9.5); в усреднение по-прежнему входит **только** `labor_units` — перенос стоимости средств производства не искажает цену живого труда.

### 11.3. Pledge (0x72) — Обещание финансирования

Публичное обещание оплатить (часть) стоимости будущей работы (`economy.md` §7). Само по себе ничего не передаёт — вес имеет только расчёт: `Transfer` с `reason` → на этот Pledge. Отзывен до расчёта (§11.4), авто-истекает по `expires`.

```
Pledge = {
  "t":         0x72,
  "target":    Ref,           // финансируемая работа/идея (Concept, Composite, WorkRecord, ...)
  "units":     Float,         // обещанные усреднённые трудочасы
  "executor":  Bytes(32)?,    // опционально: конкретная цепь-исполнитель
  "expires":   Int?,          // Unix timestamp; после — считается отозванным
  "timestamp": Int
}
```

Правила расчёта:

- **Частями**: несколько `Transfer`'ов с `reason` → Pledge, по мере приёмки этапов — постепенное выполнение, поэтапная оплата.
- **Ровно стоимость** (2026-07-10, было «не выше»): сумма всех выплат по одной работе равна сумме `labor_units` её приёмок — последний транш закрывает остаток ровно (§12.8). Превышение обещанного над стоимостью — сигнал приоритета, не цены: не выплачивается.
- **Детерминированное истечение**: расчётом считается только `Transfer` с `timestamp` < `expires`. Сравниваются два значения из цепей — гонок стенных часов нет. Порядок «отзыв против расчёта» упорядочивается через `timestamp_witnessed` (sync.md §9).

### 11.4. PledgeRevoke (0x73) — Отзыв обещания

```
PledgeRevoke = {
  "t":         0x73,
  "pledge":    Ref,           // собственный Pledge
  "timestamp": Int
}
```

Действителен только для Pledge из той же цепи и только до расчёта; при частичном расчёте отзывается неоплаченный остаток.

### 11.5. Redemption (0x74) — Квитанция погашения

Звено нити эмиссии со знаком «+» (`economy.md` §4.3): эмитент, получив собственную бумагу, фиксирует уменьшение долга. Пишется автоматически при приёме перевода со своей бумагой (`bc transfer recv`). Кредитная история цепи — максимальный закрытый долг, скорость погашения (`economy.md` §5а) — читается прямо по нити.

```
Redemption = {
  "t":          0x74,
  "transfer":   Ref,          // Transfer плательщика, вернувший бумагу
  "units":      Float,        // сколько своей бумаги аннигилировало
  "seq":        Int,          // сквозной счётчик звеньев (общий с эмиссиями)
  "prev":       Ref?,         // предыдущее звено нити
  "debt_after": Float,        // декларируемый долг после погашения
  "timestamp":  Int
}
```

Два звена с одним `seq` — эквивокация, объективное доказательство (кандидат `FraudClaim`, вскрывается при встрече ветвей). Чекпойнт `(seq, долг)` в коммите публичной вершины и правило «незасвидетельствованная эмиссия — без веса» — после реализации коммитов вершины (`economy.md` §4.3).

### 11.6. Кредитная история с нити эмиссии — слой 1 экономического следа (ИР-010)

**Реализовано (2026-07-16).** Первая витрина экономического следа: что нить эмиссии (§11.1/§11.5, `economy.md` §4.3) говорит о цепи **до** какого-либо доверия третьим лицам. Чистая функция `records::credit_history` (заголовок `records/credit.h`), потребитель — `bc trust <цепь|префикс>`.

**Считается только локально** (решение ИР-010 п. 3): звенья читаются из блоков цепи-предмета, которые спрашивающий уже держит у себя (собственные ветки — для своей цепи, `external_blocks` после слияний — для чужой). Никакого агрегатора: это личная, никем не опосредованная картина.

Метрики (все — по видимой части нити):

- **выпущено / погашено** — суммы самоэмиссий и погашений (и доля `redeemed/issued`);
- **пик долга** — самый глубокий задекларированный `debt_after`;
- **максимально возвращено с пика** («столько доверили — вернул») — наибольшее снижение долга от пика за историю (максимальная просадка кривой долга);
- **объявленный долг сейчас** — `−debt_after` последнего видимого звена;
- давность последнего погашения и последней самоэмиссии.

Правила честности (обязательные):

1. **Неполная картина — не порок цепи, а свойство наблюдателя.** Видимые звенья считаются против `max(seq)+1`; при пропусках выводится «видно N из M — картина неполная», и **осуждающие флаги не поднимаются** (пропущенным звеном могло быть погашение).
2. **Флаг «долг растёт без погашений»** — только на полной картине: ≥2 самоэмиссий и ни одного погашения.
3. **Флаг «нить не сходится»** — только на полной картине: задекларированный долг ≠ выпущено − погашено (самоаттестация нарушена).
4. **Эквивокация** — одно `seq` в двух разных блоках — сообщается всегда: это объективное доказательство, не зависящее от полноты картины. Один и тот же блок, пришедший дважды (merge + fetch), эквивокацией не является.
5. Рядом с выводом — **цена подделки** источника (подделка нити = эквивокация, ловится при первом расхождении).

## 12. Принципы трудовой экономики

1. **Единственная единица ценности** — усреднённый трудочас (`labor_units`). Никакой другой валюты нет.
2. **Трудочасы рождаются взаимным кредитом** (вексельная модель). Пара «долг/требование» возникает в момент оплаты — `Transfer` с самоэмиссией (§11.1): плательщик уходит в минус по собственной бумаге, получатель держит его именные часы. Эмитента-центра и эмиссии нет; система стартует со всеобщего нуля. `Acceptance` определяет **стоимость** работы, но часов не создаёт; до подтверждения получения и проверки качества работа не имеет стоимости.
3. **Природные ресурсы и энергия** не имеют собственной стоимости. Стоимость появляется только через труд по добыче, транспортировке и преобразованию.
4. **Отрицательный труд** — умышленное уничтожение принятой продукции. Оформляется через `NegativeWorkRecord`. Уменьшает суммарные трудочасы в системе.
5. **Коэффициент специальности** ежесуточно рассчитывается агрегаторами: суммарные часы по всем (Specialty, Grade) нормируются так, чтобы среднее = 1.0. Финальный коэффициент = медиана по всем участвующим агрегаторам.
6. **Все обмены** — только через `Transfer` в усреднённых трудочасах.
7. **Именные часы, никакой социализации.** Часы разных эмитентов не смешиваются: каждая порция несёт цепь-должника (§11.1 `origins`). Единица неизменна — час равен часу, курсов и дисконтов между бумагами нет; контрагент либо принимает бумагу данного эмитента, либо отказывает. Возврат бумаги эмитенту гасит его долг. Смерть цепи — потеря держателей именно её бумаги, а не всех: инфляция единицы невозможна по построению. Инвариант по каждой цепи: её долг равен сумме её часов на руках у других — проверяется аналитически по цепям.
8. **Оплата труда ровно равна его стоимости** (2026-07-10, было «не выше»). Выплаты по работе в сумме равны `labor_units` её приёмок: сверх труда ничего не рождается, премий нет, а недоплата исключена как скрытая скидка мимо статистики — самоэмиссия снимает ограничение ликвидности, «не могу заплатить» не существует. Вся гибкость цены — k и разряд **до** приёмки, согласуемые на каждый этап отдельно (`economy.md` §2а, §4.2). Нужность выражается **приоритетом** (объём обещаний Pledge, `economy.md` §7); срочность, опасность, вредность — будущие **измерения труда** через специальности и разряды с собственными коэффициентами (§14.8), а не надбавки к цене.
9. **Беспричинных переводов не существует** (2026-07-10). Перевод признаётся только против принятого труда — живого или овеществлённого: `reason` → `Acceptance` обязателен (Transfer v3, §11.1 — реализовано: отказ при создании и при приёме). Дарений, кредита «просто так» и прощения долга нет. Плательщик может не совпадать с благополучателем труда: приёмку пишет благополучатель, платит любой (уход за нетрудоспособными, финансирование идей — `economy.md` §4.2, §7).
10. **Долг гасится только трудом.** Погашение (`issuer == to`) признаётся только против приёмки труда эмитента — живого или овеществлённого его вещью по остаточной стоимости (`economy.md` §5б). Закрывает лазейку «скупить бумагу должника и подарить назад».
11. **Расчётность требует заверенности** (направление). Сделка входит в усреднение и кредитную историю, только если её `WorkRecord` заверён пломбами независимых цепей (`economy.md` §3, §4.2); N свидетелей — открытый вопрос.

## 13. Роль агрегаторов

Агрегатор — узел, который индексирует множество цепочек и публикует производные данные в своём блокчейне. Агрегатором может быть любой пользователь в той мере, в которой позволяют его ресурсы.

**Что вычисляет агрегатор:**

- **Реакции:** суммирует `Reaction.value` по всем копиям одного оригинала (оригинал идентифицируется по `Copy.source`). Учитывает только последнюю реакцию каждого пользователя на данный `target` (по порядковому индексу блока, не по `timestamp_claimed`).
- **Коэффициенты:** ежесуточно собирает все `WorkRecord` и `Acceptance`, вычисляет `DailyAggregate`.
- **Индекс ссылок:** хранит таблицу `(chain, hash) → location`, позволяя находить блоки без прямого обращения к владельцу.
- **Профили и потребности (§8.6):** сканирует `Concept`-ы с фасетными тегами `kind:*`, группирует по цепям и отдаёт JSON-экспорт (`GET /profiles`, `GET /profiles/<chain>`) — навыки, потребности, желания, отрасли участников. Предназначен для ручной стыковки «потребность↔навык» или для передачи внешнему ИИ (проект LLM в себя не встраивает). Как и всё у агрегатора — производная, перепроверяемая по цепям.

**Агрегаторы — кэш и индекс, не источник истины.** Любое утверждение агрегатора может быть проверено P2P-обращением к исходной цепочке.

**Масштаб агрегации:**

| Уровень | Хранит |
|---------|--------|
| Минимальный | только свою цепочку |
| Локальный | цепочки известных пользователей |
| Полный | все публичные цепочки сети |

## 14. Открытые вопросы

### 14.1. Идентификация «последней» реакции
Агрегатор учитывает только последнюю `Reaction` пользователя на конкретный `target`. Рекомендуется использовать порядковый индекс блока в ветке (монотонный, не зависит от честности автора), а не `timestamp_claimed`.

### 14.2. Весовая функция реакций
Все пользователи имеют равный вес в реакциях? Или вес реакции масштабируется по накопленным трудочасам автора? Требует явного решения — влияет на расчёт агрегаторов.

### 14.3. Порядок элементов в Composite
Порядок ссылок в `Composite.parts` влияет на хеш. Варианты: сортировать (детерминированный хеш независимо от автора) или сохранять авторский порядок (семантически значимый, но непредсказуемый хеш). Выбор влияет на дедупликацию.

### 14.4. Верификация квалификации Worker — НАПРАВЛЕНИЕ ЗАФИКСИРОВАНО (2026-07-10)
**Лестница разрядов** (`economy.md` §2а): разряд — не самозаявление, а вывод из истории цепи: обучение → экзамен (приёмка цепями, уже имеющими этот или высший разряд) → заверенный объём труда на текущем разряде → следующий разряд. Понизить разряд в сделке можно всегда, подняться — только по лестнице. Открыты детали: формат записи экзамена, пороги объёма труда на разряд.

**Пилотная механика — реализована (2026-07-15, ИР-006).** До полной лестницы действует честная середина: заявленный разряд (`grade:` в профиле) **принимается, но помечается**. Заверение = принятая работа: `Acceptance → WorkRecord → Grade → Specialty(слаг)`, каждый переход обязан оставаться в цепи труженика (чужой разряд к своей работе не пришьёшь). Считаются **разные принявшие цепи**, а не приёмки; собственная цепь не свидетель — этот само-поручительский трюк бесплатен. Справочник и стыковка показывают обе стороны: «разряд 6 (заявлен, приёмок нет)» против «разряд 4 (заверен: 2 цепями, 12 ч)» — читатель сравнивает и решает, глобальный балл не вычисляется (принципы ИР-009/ИР-010). Пустое заявление перестаёт быть бесплатным: оно видно как пустое. Заверение от нарисованных цепей остаётся возможным — ответ на это не счётчик, а поток доверия (ИР-009).

### 14.5. Пересчёт суточного агрегата — РЕШЕНО
Действует последняя запись на данную дату от данного агрегатора (закреплено в §11.2).

### 14.6. Граница между PayloadRecord и пустым Concept — РЕШЕНО (2026-07-13)
Профиль и потребности выражаются `Concept`-ами с зарезервированными тегами (§8.6), выбор записи — из универсальных каталогов (§8.7). `Entity`/`Property` остаются для структурного описания сущностей, где важны типизированные атрибуты; свободное самоописание участника — теговый `Concept`.

### 14.7. Заявка на исполнение работы — РЕШЕНО (2026-07-15, ИР-006)
Без нового типа: заявка — `ConceptLink{kind:"берусь", from: свой навык, to: потребность}` (§8.6), выбор исполнителя — существующее поле `executor` у `Pledge`. Под одну потребность может быть много «берусь»; наняли того, на кого указал `executor` (или с кем просто начали работать — обещание необязательно). Выбор по-прежнему социальный, но намерение теперь записано и видно в `DealView`.

### 14.8. Дополнительные измерения труда — РЕШЕНО
Отдельные **деятельности** со своими ставками, не надбавки и не поля (пример из `main_ideas.pdf`: «подготовка к операции» 3 стч/ч и «операция» 14 стч/ч — один хирург, разные деятельности). «Монтаж» и «монтаж на высоте» — разные специальности в статистике §11.2. Принцип §12.8 не тронут.
